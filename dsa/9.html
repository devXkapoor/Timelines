Era IX · Characters, Words, and Patterns

Strings: Pattern Matching, Tries, and the Algorithms That Read Text

Text is everywhere — and "find this word in that document" turns out to be a much more interesting algorithmic problem than it first appears, with connections to automata theory, suffix structures, and one of the most elegant "avoid redoing work" ideas in all of computer science.

The shift in this era. Until now, the building blocks of problems have mostly been numbers — values to sort, distances to minimize, counts to maximize. This era turns to a different kind of data: strings, which are sequences of characters (letters, digits, symbols). Text processing is arguably the most universal computing task in existence — every search engine, every text editor, every compiler, every biological sequence database, every spam filter works fundamentally on strings.

Strings look simple but create subtle, interesting problems. "Does this document contain the word 'algorithm'?" sounds trivial — but what if the document is a billion characters long and you need to search it millions of times? What if you're searching for all occurrences of a pattern, not just the first? What if you want to find the longest portion two strings have in common? Each question leads to a distinct, carefully-designed algorithm, and together they form one of the richest sub-areas of DSA — and one of the most consistently-tested in competitive programming.

The baseline — and why it falls short

Naïve Pattern Matching — and the O(n·m) Problem

+
The Setup
Given a "text" string of length n and a "pattern" string of length m, find all positions in the text where the pattern appears. The naïve approach: try aligning the pattern at every possible position in the text (positions 0, 1, 2, ..., n−m), and at each position, compare the pattern to the text character by character until either a full match is confirmed or a mismatch stops the comparison early.
Vocabulary
Stringa sequence of characters. "Hello" is a string of length 5. Characters are usually drawn from some "alphabet" — the set of possible characters — which might be just the 26 letters, or all ASCII characters, or the four DNA bases, etc.
Pattern / textin string matching, "text" is the longer string being searched in, "pattern" is the shorter string being searched for. Often written as T (length n) and P (length m), with m ≤ n typically.
Substringa contiguous portion of a string — characters at positions i through j (inclusive), for some i ≤ j. "algo" is a substring of "algorithm" (positions 0–3).
Alignment / shiftpositioning the pattern at a specific starting offset in the text, to check whether the text matches the pattern starting at that position.
First Principles

The naïve algorithm is exactly what most people would write intuitively: try every possible starting position in the text, and at each one, check character by character whether the pattern fits.

At shift 2, the naïve algorithm matched 4 characters before seeing a mismatch. It then moves to shift 3 — but doesn't use what it just learned (that the text contains "ABAB" starting at position 2). KMP, the next entry, does.

The naïve algorithm is correct and perfectly fine for small inputs. Its complexity: at each of the up to (n − m + 1) starting positions, up to m character comparisons in the worst case. Total: O(n × m). For typical inputs, mismatches usually happen early at each position, making the practical behavior much better. But in the worst case — specifically when the text and pattern consist of many repeated characters (e.g., text = "AAAAAAAAAB", pattern = "AAAAB") — every alignment almost fully matches before finally finding the mismatch at the last character. This is the adversarial input for naïve matching, and it makes O(n × m) a real concern: for n = 100,000 and m = 10,000, that's 1 billion comparisons.

The deeper observation: naïve matching throws away information. After finding that text[2..5] = "ABAB" matches pattern[0..3] = "ABAB" before a mismatch at position 4, the naïve algorithm forgets everything it learned and starts fresh at position 3. But it could have observed: "I just confirmed that text[2..5] = ABAB. I know (from the pattern itself) that ABAB starts with AB — so text[4..5] = AB already, and my next alignment at shift 4 can skip those first two character comparisons." This is the key insight that the next entry formalizes.

Why this matters O(n × m) is the baseline to beat. For bioinformatics (searching a human genome of ~3 billion base pairs for a gene sequence of thousands of bases), O(n × m) is completely intractable — you need something better. For competitive programming, text problems with n and m up to 10^5 or 10^6 typically require an O(n + m) or O(n log n) solution — and the algorithms in this era are what provide those.
1977 · Donald Knuth, James Morris, Vaughan Pratt · USA

The Knuth-Morris-Pratt Algorithm — Never Re-examine a Character

+
The Source
KMP achieves O(n + m) string matching by precomputing a "failure function" (also called the "prefix function") for the pattern alone — a table that, for each position in the pattern where a mismatch might occur, records how far back in the pattern the search can resume without re-examining already-matched text characters.
Vocabulary
Prefixa substring starting at position 0 — the beginning of a string. Prefixes of "ABAB" are "", "A", "AB", "ABA", "ABAB".
Suffixa substring ending at the last position — the end of a string. Suffixes of "ABAB" are "", "B", "AB", "BAB", "ABAB".
Proper prefix / suffixa prefix or suffix that is strictly shorter than the whole string (i.e., not the string itself).
Failure function / prefix functionfor the pattern P, an array where entry π[i] = the length of the longest proper prefix of P[0..i] that is also a suffix of P[0..i]. This encodes, for every position in the pattern, "how much of the pattern have I already confirmed, even if a mismatch occurs here?"
Automatona mathematical model of a machine with a finite number of "states" and rules for transitioning between states based on input. KMP can be understood as building and running a tiny automaton for the pattern — connecting it directly to Turing's "states" from Era II.
First Principles

The KMP algorithm's central question: when a mismatch occurs at position j in the pattern (after successfully matching positions 0 through j−1), instead of naïvely shifting the pattern by just 1 and restarting from the beginning, how far can we safely skip — and where in the pattern can we resume — without risking missing a match?

The answer comes from examining the pattern itself, not the text. Here's why: we just confirmed that text[i−j .. i−1] = pattern[0 .. j−1]. If pattern[0..j−1] has a proper prefix that also equals a proper suffix of pattern[0..j−1], say of length k, then text[i−k .. i−1] = pattern[0..k−1] (the last k text characters we already matched also match the first k characters of the pattern). So instead of shifting to position i−j+1 and restarting from j=0, we can shift to position i−k and resume from j=k — we've already "confirmed" those first k characters, without re-reading any text.

π[3]=2 means: if a mismatch occurs at pattern position 4 (after matching "ABAB"), the longest prefix of "ABAB" that is also a suffix of "ABAB" has length 2 — "AB". So resume matching from pattern position 2, having already confirmed "AB" in the text. The text pointer never moves backward.

The failure function π is computed in O(m) by the same matching logic applied to the pattern matching against itself — an elegant recursion. The matching phase uses this table to skip forward efficiently: the text pointer never moves left — it only ever advances forward, one character at a time. The pattern pointer may jump back (via π), but each jump is strictly smaller than the text pointer's total advancement. This is the key insight for the complexity proof: the text pointer makes exactly n forward steps total, and no step is ever repeated — so the total number of character comparisons across all alignments is O(n), plus O(m) for precomputing π, giving the celebrated O(n + m) overall — regardless of input content, including the adversarial cases that destroy naïve matching.

There is a beautiful connection to automata theory (Era II). The failure function effectively encodes a finite-state automaton for the pattern — each "state" represents how many characters of the pattern have been matched so far, transitions on matching characters advance the state, and transitions on mismatches jump to the state encoded by π (the "longest matching prefix we can salvage"). Running the text through this automaton, one character at a time, is exactly KMP matching. The state machine connection explains why KMP can't miss matches: it processes every text character exactly once and never backtracks, because the automaton's transition function (computed from π) guarantees correct behavior at every character.

Why this matters for CP KMP appears directly in problems asking for occurrences of a pattern in a text. But the failure function / prefix function is arguably even more useful as a standalone tool — it's used in "Z-function" problems (a related, often simpler-to-implement concept), and both appear constantly in string problems that aren't obviously "find this pattern here" but reduce to it. The deeper skill: recognizing that "this string problem involves some repeated prefix/suffix structure" → think KMP or Z-function.

The Trie — A Tree Shaped Like the Words It Stores

+
The Source
A trie (from "retrieval") stores a set of strings by representing them as paths through a tree — the root represents the empty string, each edge is labeled with a character, and each node represents the string formed by following edges from the root to that node. Two strings sharing a prefix share the corresponding path from the root.
Vocabulary
Triepronounced "try" (rhymes with "try," not "tree," though people often say "tree" too — both are heard). A tree where the path from root to a node spells out the prefix represented by that node.
Prefix sharingif two strings in a trie share a prefix (e.g., "ant" and "any" share "an"), they share the same path in the trie up to where they diverge. This shared-path property is the trie's defining structural feature.
End-of-word markera flag or special node indicating that the string formed by the path from root to this node is a complete, stored word — distinguishing a word from a mere prefix of another stored word (e.g., "an" is a prefix of "ant," but if "an" is also a stored word, it needs a marker).
Autocompletegiven a prefix typed by a user, finding all words that complete it — a natural trie operation: descend from the root following the prefix's characters, then collect all words in the subtree below that node.
First Principles

Suppose you're storing a dictionary of English words and want to support: "does this word exist?", "does any stored word start with this prefix?", and "list all words starting with this prefix." Hash tables (Era VIII) can answer "does this word exist?" in O(1) average, but can't naturally answer prefix-related questions (hashing destroys the prefix structure). Binary search trees (Era IV/VIII) can, but searching a BST for a word of length m requires up to O(log n) comparisons, each of which may require comparing two entire strings (which costs O(m) in the worst case) — so BST lookup is O(m × log n). A trie gives you O(m) for all three queries — proportional only to the length of the word or prefix, completely independent of how many words are stored.

"ant," "any," and "an" all share the path a→n from the root — their common prefix. Looking up any of these means following edges labeled by each character, one per step. The ✓ marks end-of-word nodes. Finding all words starting with "an" means descending to the "an" node and collecting all ✓ nodes below it.

The trie's O(m) lookup doesn't depend on n (the number of stored words) at all, making it ideal for large vocabularies. The cost: memory. A trie node needs space for pointers to each possible child character — for the English alphabet (26 letters), each node holds 26 pointers. If most words are short and share few prefixes, most of these pointers are empty (null), and the memory use is O(total characters across all stored words) but with a potentially large constant. "Compressed tries" (also called Patricia trees or radix trees) address this by collapsing runs of single-child nodes into single edges labeled by the entire substring, reducing memory at the cost of slightly more complex code.

A particularly important application in CP: the XOR trie (also called "binary trie"). Numbers can be treated as binary strings (sequences of bits), and a trie on these bit-strings can answer "what stored number has the maximum XOR with a given query number?" in O(32) or O(64) steps (one per bit) — much faster than scanning all stored numbers. This technique appears in a notable category of competitive programming problems involving bitwise operations on sets of numbers.

Why this matters for CP Tries appear in CP problems involving prefix matching, autocomplete-style queries, IP routing, and — importantly — as the structure underlying more advanced string algorithms. They're also, in a broader sense, a beautiful example of the "shape of the data encodes the structure of the problem" design principle: the trie's branching structure exactly mirrors the prefix-tree structure of the strings it stores, which is why prefix operations are "free" — they're just tree traversals.
1970s · Multiple contributors; edit distance by Levenshtein 1965

Comparing Strings: LCS and Edit Distance via Dynamic Programming

+
The Source
Given two strings, the "Longest Common Subsequence" (LCS) problem asks for the longest sequence of characters appearing in both strings in order (but not necessarily contiguously). The "edit distance" (Levenshtein distance) asks for the minimum number of insertions, deletions, or substitutions needed to transform one string into the other. Both are solved via a 2D DP table, each a direct application of Era VII's principles to string data.
Vocabulary
Subsequencea sequence derived from another string by deleting some (or no) characters without changing the order of the remaining characters. "ACE" is a subsequence of "ABCDE" — pick characters at positions 0, 2, 4. Note: unlike a substring, a subsequence need not be contiguous.
LCSLongest Common Subsequence — the longest sequence of characters (in order, not necessarily contiguous) that appears in both strings. For "ABCBDAB" and "BDCAB", the LCS is "BCAB" or "BDAB" (both length 4).
Edit distancethe minimum number of single-character edits (insert, delete, substitute) to convert one string into another. Also called Levenshtein distance. "kitten" → "sitting" has edit distance 3.
2D DP tablea table with one dimension for each string's index, where cell (i, j) stores the answer to the subproblem "considering only the first i characters of string A and the first j characters of string B." Filling this table left-to-right, top-to-bottom, using previously-filled cells, solves the full problem.
First Principles

These two problems are the canonical examples of DP applied to strings, and they share the same template — a 2D table where both dimensions are string indices — even though they model quite different questions. They're the bridge from "DP on numbers" (Era VII's Fibonacci, paths, counts) to "DP on text," and they demonstrate that the same framework handles both kinds of data naturally.

For LCS, the recurrence is: LCS(i, j) = the length of the LCS of the first i characters of string A and the first j characters of string B. If A[i] == B[j] (the current characters match), this common character is part of the LCS: LCS(i,j) = LCS(i−1, j−1) + 1. If they don't match, we can't use both — we skip one or the other and take whichever gives longer LCS: LCS(i,j) = max(LCS(i−1,j), LCS(i,j−1)).

For edit distance, the recurrence is similar but encodes a different kind of "choice." At each cell (i, j): if A[i] == B[j], no edit needed here, so edit(i,j) = edit(i−1, j−1). If they differ, we choose the cheapest of three operations: substituting A[i] to match B[j] (edit(i−1,j−1) + 1), inserting B[j] into A (edit(i, j−1) + 1), or deleting A[i] (edit(i−1, j) + 1), and take the minimum.

Both problems are O(n × m) in time and space, filling a table of that size — usually manageable for n, m up to a few thousand, but requiring the rolling space optimization from Era VII (keeping only two rows at a time) to handle larger inputs when only the final answer is needed rather than the full table.

Beyond competitive programming, edit distance is the core of spell-checkers, DNA sequence alignment, plagiarism detection, and "diff" tools (the program that shows what changed between two versions of a file). LCS underlies the Unix diff command, version control systems, and bioinformatics. These two problems connect Era VII's abstract DP framework to one of the most practically impactful areas of computer science.

The full picture of Era IX This era introduced strings as a data type with their own rich algorithmic sub-area: the naïve O(n·m) baseline and why it fails on adversarial inputs, KMP's O(n+m) guarantee via the failure function (and its elegant automaton interpretation connecting back to Era II), the trie as a tree structure that makes prefix operations natural and fast, and LCS/edit distance as Era VII's 2D DP applied to text comparison. These four ideas cover the most common "string category" problems in competitive programming and form the foundation for the more advanced string structures (suffix arrays, suffix automata) that appear at higher levels of competition.
