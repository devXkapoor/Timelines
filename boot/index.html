Power Button Pressed — The Signal Chain — T+0ms

The section text

"Voltage reaches the motherboard. The Power Supply Unit negotiates with the chipset. Nothing runs until the PSU says it is ready."

And the expanded detail from the timeline:

"When you press the power button, you are closing a physical switch on the motherboard. This switch connects a specific motherboard pin — called PWR_BTN# — to ground, pulling it low. The motherboard's chipset detects this signal and sends a PS_ON# signal to the PSU. The PSU begins powering up its internal circuits. It does NOT immediately send full power — it first ensures all its output voltages are stable. When ready, it asserts PWR_OK (Power Good). The time between pressing the button and PWR_OK going high: typically 100–500 milliseconds. During this time, the CPU is held in a reset state."

Every term, explained from absolute zero

Voltage — A measure of electrical potential difference between two points. Think of it like water pressure in a pipe — voltage is the "pressure" that pushes electrons through a conductor. Measured in volts (V). High voltage means more "pressure" pushing electrons. Zero voltage means no pressure — no current flows. In digital electronics, specific voltage ranges represent binary 0 and 1 — typically 0–0.8V means 0, and 2V–3.3V (or 2V–5V depending on the generation) means 1.

Motherboard — The main circuit board inside a computer — a large, flat, rigid board made of fiberglass with copper traces (wires) etched into it, onto which all the major components are attached or plugged. The CPU sits in a socket on the motherboard. RAM plugs into slots on the motherboard. The hard drive connects via cables to the motherboard. The motherboard is the backbone that physically and electrically connects all components together.

Physical switch — A mechanical device that either completes or breaks an electrical circuit. A light switch is a physical switch. When it is in the OFF position, a gap exists in the circuit — no current flows. When switched ON, the gap closes — current flows. The power button on your computer case is a physical switch connected by two wires to two pins on the motherboard.

PWR_BTN# — Power Button signal — A specific electrical signal line on the motherboard. The # symbol (sometimes written with a line over the letters) is electronics notation meaning "active low" — this signal is active (meaning "power button pressed") when the voltage on this line is LOW (near 0V), not when it is high. When you press the power button, you connect this line to electrical ground (0V), pulling it low — asserting the signal. When you release the button, the line returns to a higher voltage (pulled up by a resistor inside the chipset) — deasserting the signal. This is a momentary press — you press and release, and the chipset registers the event.

Ground — The reference point in an electrical circuit — the baseline voltage from which all other voltages are measured. Defined as 0 volts by convention. In a computer, the metal chassis, the power supply's negative terminal, and specific pins on every chip are all connected to ground. When we say "pulling a signal to ground" or "pulling it low," we mean connecting that signal line to the 0V reference, which forces its voltage to 0V.

Resistor — An electronic component that resists the flow of current. A "pull-up resistor" connects a signal line to a positive voltage (say, 3.3V) through a resistor. Under normal conditions (nothing else driving the line), the resistor pulls the voltage up to 3.3V. When something else connects the line to ground, current flows through the resistor (controlled by its resistance), and the line's voltage drops to near 0V. This is how "active low" signals work — the line sits at high voltage normally, and is pulled low to assert the signal.

Chipset — A chip (or set of chips) on the motherboard that manages communication between the CPU, RAM, storage devices, and peripheral devices. On modern Intel systems, this is called the PCH (Platform Controller Hub). On AMD systems, it's called the Fusion Controller Hub or similar. The chipset handles things the CPU delegates — USB control, PCIe lane routing, SATA storage, audio, and — relevant here — power management and reset sequencing. The chipset is always powered (from the 5VSB standby rail, explained shortly) even when the computer appears "off."

PSU — Power Supply Unit — The component that converts AC (Alternating Current) from your wall outlet into the DC (Direct Current) voltages computers need. Your wall outlet provides 110V AC (in North America) or 220V AC (in Europe) — alternating current that reverses direction 50 or 60 times per second. Computer components need stable, low-voltage DC: +12V, +5V, +3.3V. The PSU contains a transformer (steps down the voltage), rectifiers (convert AC to DC), filtering capacitors (smooth out ripple), and voltage regulators (maintain stable outputs despite varying load). It connects to the motherboard via a large multi-pin connector.

AC — Alternating Current — Electrical current that periodically reverses direction. The voltage alternates between positive and negative values in a sine wave pattern. Grid power is AC because it is more efficient to transmit over long distances at high voltage using transformers. Computer components cannot use AC directly — they need DC.

DC — Direct Current — Electrical current that flows in one direction only, at a constant voltage. Batteries produce DC. Computer chips need DC because they store bits as charges in capacitors or flip-flops — a stable voltage is needed to reliably distinguish 0 from 1.

PS_ON# — Power Supply On signal — A signal line between the motherboard and the PSU. When the chipset wants to turn on the PSU's main outputs, it pulls this line low (active low, per the # notation). The PSU monitors this line. When PS_ON# goes low, the PSU starts its power-up sequence. When PS_ON# goes high (deasserted), the PSU shuts off its main outputs. This is why your computer can be "off" but the PSU is still plugged in and the 5VSB rail is active — PS_ON# is high (deasserted), main outputs are off, but the PSU is waiting.

PWR_OK — Power Good signal — A signal line from the PSU back to the motherboard. When all the PSU's output voltages have reached their correct stable levels and remained stable for a minimum time (typically 100ms), the PSU asserts PWR_OK — drives this line to a defined high voltage. This is the PSU's declaration: "my outputs are valid, it is safe to start the electronics." The chipset monitors PWR_OK and does not release the CPU's reset signal until PWR_OK has been asserted.

Reset state — A special condition the CPU is in from the moment it receives power until the reset signal is released. In reset state, the CPU is powered (transistors have voltage) but its internal state machine is held in a fixed, defined configuration by the RESET# signal being asserted (held low). The CPU executes no instructions. When RESET# is released (goes high), the CPU's internal state machine initializes all registers to their documented reset values and begins executing — we covered these values in the CPU architecture documents.

RESET# pin — A physical pin on the CPU chip package connected to a corresponding signal from the chipset. While RESET# is held at low voltage by the chipset, the CPU is in reset — all its internal state is frozen at initial values. When the chipset releases RESET# (allows it to go high), the CPU begins operation. The chipset keeps RESET# asserted until PWR_OK arrives, ensuring the CPU never tries to execute with unstable power.

5VSB — 5 Volt Standby — A special always-on output rail from the PSU that provides +5V even when the computer is "turned off" (PS_ON# deasserted, main outputs off). This rail powers:

  • The chipset's power management circuitry (so it can monitor the power button)
  • The RTC (Real-Time Clock) — the clock that keeps time when the computer is off
  • Wake-on-LAN and Wake-on-USB circuitry
  • USB ports marked as "always on" for charging

This is why your phone charges from a USB port even when the computer is off, and why unplugging the computer from the wall causes it to completely lose power (including the ability to respond to the power button).

Capacitor — An electronic component that stores electrical charge. A capacitor consists of two conductive plates separated by an insulating material. When voltage is applied, charge accumulates on the plates. The capacitor can then release this stored charge as current. In PSUs, large capacitors smooth out voltage fluctuations — when the AC input briefly dips, the capacitors release stored charge to maintain stable DC output. The size of the capacitors determines how long the PSU can maintain stable output through brief power interruptions.

Voltage regulator — A circuit that maintains a stable output voltage despite variations in input voltage or output current (load). Inside the PSU, separate voltage regulators maintain each output rail (+12V, +5V, +3.3V) at their specified values. On modern motherboards, there are additional voltage regulator modules (VRMs) near the CPU socket that further step down the PSU's +12V to the much lower voltages the CPU actually needs (modern CPUs run at 0.7V–1.4V depending on workload and clock speed).

Transformer — An electromagnetic device that converts AC voltage from one level to another. A transformer has two wire coils wound around an iron core. AC current in the primary coil creates a changing magnetic field in the core, which induces AC current in the secondary coil. The ratio of the number of wire turns in each coil determines the voltage ratio. A transformer with 10× fewer turns in the secondary coil outputs 1/10th the voltage of the input. This is how 220V AC from the wall becomes lower-voltage AC inside the PSU before conversion to DC.

Rectifier — A circuit that converts AC (which alternates between positive and negative) into DC (which flows in one direction). A rectifier uses diodes — components that allow current to flow in only one direction. By arranging four diodes in a "bridge" configuration, both the positive and negative halves of the AC sine wave are converted to positive DC. The result is not perfectly smooth DC — it has a ripple (pulses at twice the AC frequency) that the capacitors then smooth out.

What actually happened — from first principles

Why power-up sequencing exists at all

A naive view of "turning on a computer" is: connect power, everything starts simultaneously. But this cannot work.

Consider: the CPU needs to execute instructions from memory. But memory (RAM) requires initialization before it contains valid data. The CPU needs to know about the memory controller settings. The memory controller is part of the chipset. The chipset needs stable voltage before it can operate. Stable voltage requires the PSU to have completed its power-up sequence. The PSU needs the PS_ON# signal from the chipset. But the chipset needs stable 5VSB to even monitor the power button.

These dependencies form a chain. Each component depends on others being ready first. Power-up sequencing solves this by ensuring things happen in the correct order, with each step gating the next.

The physical sequence, traced through silicon and copper

T = 0ms: The button press

You press the power button. The button is a momentary normally-open switch — when not pressed, no contact is made; when pressed, two electrical contacts touch.

Inside the computer case, two thin wires run from the power button to two pins on the motherboard — a header connector labeled "PWR_BTN." When the button is pressed, these two pins are connected together. One pin is connected (inside the chipset) to a pull-up resistor that holds it at 3.3V under normal conditions. The other pin is connected to ground.

When the button connects them: the first pin is now connected to ground through the button, and the pull-up resistor limits the current flow. The voltage on PWR_BTN# drops from 3.3V to near 0V — the signal is asserted.

Inside the chipset — detecting the button press

The chipset contains logic that monitors PWR_BTN# continuously, even when the main computer is off, because the chipset's power management circuits are powered by 5VSB. This logic detects the falling edge (transition from high to low voltage) on PWR_BTN#. To prevent accidental triggers from button contact bounce (the mechanical contacts briefly bounce when they touch, creating multiple rapid transitions), the chipset implements debouncing — requiring the signal to remain low for a minimum time (typically 4ms) before treating it as a valid press.

After the debounce period confirms a real button press, the chipset's power management state machine transitions from "S5" (soft off) to "S0" (fully on). Part of this transition: the chipset asserts PS_ON# — pulls that line low.

The PSU responds to PS_ON#

The PSU's control circuitry continuously monitors PS_ON# on its motherboard connector. When PS_ON# goes low, the PSU's control chip begins the power-up sequence:

  1. It enables the main switching circuits — the transistors that chop the rectified AC into pulses that the transformer can convert.
  2. The +12V rail begins rising. The control chip monitors it via a feedback loop.
  3. The +5V rail begins rising (derived from the +12V in most modern designs).
  4. The +3.3V rail begins rising.
  5. The control chip monitors all rails continuously, checking that they are within specification (+/- 5% for most rails).
  6. Once all rails have been within spec for a minimum dwell time (typically 100ms), the control chip asserts PWR_OK — drives that line to +5V.

The 100–500ms delay we mentioned: the lower end (100ms) is the minimum dwell time before PWR_OK. The upper end (500ms) accounts for PSU designs that are more conservative, caps that take longer to charge, or complex sequences with more checking steps.

Why stable voltage matters so much

A CPU register is a collection of flip-flops — circuits that hold a bit as a voltage level. If the supply voltage is slightly below spec, a flip-flop that should read as "1" might be ambiguous — it might read as "0" on the next clock cycle. The result is silent data corruption: the CPU computes wrong answers without knowing it, producing results that look like software bugs but are actually hardware failures caused by marginal voltage.

This is why PSU quality matters enormously, and why cheap PSUs that cannot maintain stable voltages under load cause mysterious crashes and corruption.

PWR_OK arrives at the motherboard

The chipset monitors PWR_OK. When it transitions from low to high, the chipset's power-on state machine proceeds: it releases RESET# — allows the CPU's reset pin to go high.

The CPU's internal RESET# monitoring circuitry detects this rising edge. On the next positive edge of the clock signal, the CPU's internal state machine initializes. Every register is forced to its documented reset value. The CPU begins its fetch-decode-execute cycle from the reset vector.

The voltage rails and why each exists

The PSU provides multiple voltage rails because different components have different requirements:

+12V is the highest voltage and carries the most power (power = voltage × current; for a given power delivery, higher voltage means lower current, which means thinner wires can be used). The +12V rail powers:

  • The CPU's local voltage regulators (which step it down to ~1V)
  • PCIe graphics card power (which also has its own local regulators)
  • Hard drive motors and read/write actuators (which need 12V directly)
  • Cooling fans

+5V historically powered most logic circuits. Early CPUs ran at 5V (the 8086, 8088, 286, 386 all used 5V logic). As transistors shrank, the safe operating voltage dropped — 3.3V, then 2.5V, then 1.8V, then 1.2V, then below 1V. Modern CPUs no longer use 5V directly, but +5V is still used for USB ports and some legacy components.

+3.3V is the current standard logic voltage for most chips that don't need dedicated VRMs. Modern DDR5 RAM uses 1.1V (with its own regulators), but older DDR4 used 1.2V (regulated from 3.3V). Many motherboard chips and slots operate at 3.3V.

+5VSB is the standby rail — always on, providing just enough power to keep the power management circuitry alive. It is connected directly to mains power through the PSU's standby supply, bypassing the main switching circuits that PS_ON# controls.

The significance of the 100–500ms delay

You might wonder: if modern CPUs run at 3–5 GHz, executing billions of instructions per second, why does power-up take hundreds of milliseconds?

The answer is that the delay is entirely in the analog electrical domain — the charging of capacitors, the stabilization of switching regulators, the thermal equilibration of transformer cores. These are governed by physics (RC time constants, electromagnetic induction) that do not benefit from digital circuit speedups. Making the capacitors charge faster would require either larger current capacity (more expensive PSU) or smaller capacitors (less filtering, more ripple, less stability). The 100–500ms is a design choice balancing cost, stability, and speed.

Once PWR_OK arrives and the CPU begins, the digital domain takes over — and things happen at clock speed. The next phase (POST) takes longer not because of analog delays but because of the sheer amount of hardware that must be tested and initialized.

CPU Reset State — All Registers to Known Values — T+~100–500ms

The section text

"PWR_OK arrives. The chipset releases the CPU's reset pin. The CPU's internal state snaps to fixed, defined values. It is ready to fetch its first instruction."

Every term, explained from absolute zero

RESET# pin — Already defined above. The physical pin on the CPU package through which the chipset holds the CPU in reset (by keeping this pin at low voltage) and releases it (by allowing it to return to high voltage).

Internal state machine — The collection of circuits inside the CPU that determines its current operating condition — what mode it is in, what phase of the fetch-decode-execute cycle it is in, what values its registers hold. A state machine is a system that can be in one of a finite number of states and transitions between them based on inputs and the current state. The CPU's internal state machine, when RESET# goes high, forces the CPU into a specific initial state regardless of whatever random state the transistors might have been in.

0x0000FFF0 — The value in EIP after reset — EIP is the instruction pointer (called IP in 16-bit mode, EIP in 32-bit mode). After reset, it holds 0x0000FFF0. This is a 32-bit hexadecimal number. In decimal: 65,520. This is an offset, not a physical address — the physical address comes from adding the CS base to this offset.

0xFFFF0000 — The CS base value after reset — The Code Segment register has a hidden portion (the "shadow" portion, covered in our 8086 and GDT documents). After reset, even though the CS selector value is 0xF000, the CS base is pre-initialized to 0xFFFF0000. This is a special initial condition — in normal real-mode operation, loading CS with selector 0xF000 would give a base of 0xF000 × 16 = 0xF0000. But at reset, Intel pre-loads the CS shadow with a base of 0xFFFF0000 rather than the value the real-mode formula would give. This is implemented in hardware specifically to place the reset vector at the top of the 32-bit address space.

0xFFFFFFF0 — The reset vector (physical address) — The physical address where the CPU fetches its first instruction after reset. Computed as: CS.base (0xFFFF0000) + EIP (0x0000FFF0) = 0xFFFFFFF0. In decimal: 4,294,967,280. This is 16 bytes below the top of the 4GB address space (the top being 0xFFFFFFFF = 4,294,967,295).

Real mode — We covered this completely in the protected mode document. The CPU's initial operating mode after reset, with segment×16+offset addressing, no memory protection, and direct physical address generation. All x86 CPUs boot in real mode.

PE bit — The Protection Enable bit — bit 0 of CR0. After reset, CR0 = 0x60000010. Bit 0 is 0, so PE = 0, meaning real mode is active. The CPU will remain in real mode until the bootloader explicitly sets this bit.

PG bit — The Paging bit — bit 31 of CR0. After reset, this is 0, meaning paging is disabled. Physical addresses equal linear addresses. Paging will not be enabled until the OS kernel explicitly enables it.

EFLAGS = 0x00000002 — The FLAGS register's initial value. Only bit 1 is set. Bit 1 is a reserved bit that Intel documentation says must always be 1 — it has no functional meaning but is always 1 by specification. All meaningful flag bits (CF, ZF, SF, OF, IF, DF, TF) start at 0. Notably, IF = 0 means interrupts are disabled — the CPU will not respond to hardware interrupts until the bootloader or OS explicitly executes the STI instruction.

CMOS — Complementary Metal-Oxide-Semiconductor — a type of chip manufacturing process that combines two types of transistors (n-type and p-type) to create logic that consumes very little power when not switching. CMOS circuits maintain their state with almost no current flow, which is why a small lithium battery on the motherboard (the CMOS battery) can keep BIOS settings and the real-time clock running for years. When people say "clear the CMOS," they mean removing this battery (or shorting its contacts) to let the small capacitors discharge, losing the stored BIOS settings.

RTC — Real-Time Clock — A dedicated low-power circuit on the motherboard (integrated into the chipset on modern boards) that keeps track of the current date and time even when the computer is off, powered by the CMOS battery. The RTC has its own oscillator (a 32.768 kHz crystal — this specific frequency is chosen because 32,768 = 2¹⁵, making it easy to divide down to exactly 1Hz for a seconds counter using simple binary dividers). The firmware reads the RTC at boot to know the current date and time.

What actually happened — from first principles

The moment RESET# goes high

The transition of RESET# from low (0V) to high (3.3V) is detected by circuits inside the CPU that monitor this pin. On the next rising edge of the CPU's clock signal, these circuits trigger a chip-wide initialization sequence.

This initialization is not software — it is purely hardware. A dedicated reset circuit, implemented in transistors, forces every register, every internal state bit, every pipeline stage register to specific values simultaneously. The entire process completes within a single clock cycle (or a small fixed number of cycles, depending on the CPU's internal complexity).

From outside the chip, the observable result is: the CPU begins driving the address bus with the reset vector address (0xFFFFFFF0) and asserts its READ control signal. This happens automatically, mandated by hardware.

Why the reset vector is at 0xFFFFFFF0 specifically

The placement of the reset vector at 0xFFFFFFF0 is the result of a specific engineering requirement: the firmware ROM must be accessible at the reset vector address, but it must also be accessible via the standard real-mode address 0xF0000 (for BIOS service calls during normal operation).

This is solved by the motherboard's address decoder doing something clever: it maps the firmware ROM to respond to addresses in TWO ranges:

  • 0x000F0000 – 0x000FFFFF (the real-mode-accessible range, where BIOS services live)
  • 0xFFFE0000 – 0xFFFFFFFF (the top of the 32-bit address space, where the reset vector lives)

Both address ranges decode to the same physical ROM chip — they are aliases. When the CPU issues address 0xFFFFFFF0, the ROM responds. When real-mode code issues address 0xF0000 + offset (to call BIOS services), the same ROM chip responds with the same data.

This aliasing is implemented in the chipset's address decoding logic — a combinational circuit that examines the address bus and decides which component should respond. For the firmware ROM, the chipset's decoder is wired to activate the ROM's chip-select for both address ranges.

What CR0's reset value tells us

CR0 = 0x60000010 after reset. Let us decode the bits that matter:

Bit 0 (PE) = 0: Real mode. The segmentation unit uses segment×16+offset, not GDT-based protection.

Bit 4 = 1: ET (Extension Type) — a historical flag from the 386 era indicating that the math coprocessor (FPU) uses the "387" protocol rather than "287." Always 1 on modern CPUs. Not functionally relevant for OS development.

Bit 29 = 1: NW (Not Write-through) — cache write-through disabled.

Bit 30 = 1: CD (Cache Disable) — the CPU's caches are disabled at reset. This is important: the caches start disabled because the cache initialization process itself requires running instructions, which requires a known-good CPU state. The firmware enables caches as part of its early initialization sequence.

Bit 31 (PG) = 0: Paging disabled. Virtual addresses equal physical addresses.

The fact that caches are disabled at reset (CD=1) means the CPU's very first instructions — including the jump from the reset vector to the firmware body — execute directly from the ROM chip without caching. This ensures the first instructions work correctly regardless of cache state. The firmware later initializes the cache (writing zeros to all cache lines via a special instruction sequence called "cache as RAM" or CAR initialization on some systems) before enabling it.

Why interrupts start disabled

EFLAGS.IF = 0 means hardware interrupts are disabled. This is essential.

The IDT (Interrupt Descriptor Table) has not been set up yet. No ISRs have been installed. If a hardware interrupt fired at this point — say, the timer chip asserting IRQ0 — the CPU would look up vector 8 (the mapped vector for IRQ0 in the IBM PC's standard PIC configuration, before remapping) in the IDT. But the IDT does not exist yet. The IDTR points to address 0 (its reset value), and whatever bytes happen to be at address 0 in ROM or memory are not valid IDT entries. The result would be a triple fault (CPU attempts to handle a double fault, fails, and resets).

By starting with interrupts disabled, the firmware and bootloader have safe time to set up their interrupt infrastructure before hardware events start firing.

The special nature of the CS shadow after reset

We must be precise about something subtle here, because it is directly relevant to understanding real mode.

After reset, CS selector = 0xF000, CS base = 0xFFFF0000 (a value that does NOT follow the real-mode formula of CS × 16). This is a special hardware initialization — the CS base is hardwired to 0xFFFF0000 at reset, not computed by the real-mode formula.

After the firmware executes its first far jump (from the reset vector to somewhere in the 0xF0000 region), CS is loaded with a new value. From that point, CS obeys the normal real-mode rule: base = CS × 16. Loading CS with 0xF000 gives base = 0xF0000 × 1 = 0xF0000 × 16... wait — in real mode, CS holds a segment value, and the base = CS × 16. So CS = 0xF000 gives base = 0xF000 × 16 = 0xF0000.

So after the first far jump: CS = 0xF000, base = 0x000F0000. The firmware now executes from the normal real-mode BIOS region at 0xF0000 and above.

This is why the BIOS shows up at both 0xF0000 (in the real-mode-accessible first 1MB) and at 0xFFFFFFF0 (at the top of the 4GB space, aliased to the same ROM) — two different addresses, one physical chip, accessed at different moments in the boot sequence.

POST — Power-On Self-Test — T+~500–600ms

The section text

"The firmware tests the CPU, counts and tests RAM, identifies installed hardware, and initializes everything that must work before the OS loads. If something fails, POST stops and reports an error."

Every term, explained from absolute zero

POST — Power-On Self-Test — The hardware verification and initialization sequence the firmware performs on every boot before searching for a bootable device. The word "self-test" reflects that the computer is testing itself — not a program the user ran, but a built-in diagnostic routine that runs automatically.

Firmware — Software that is permanently stored in a non-volatile memory chip (ROM or flash memory) on a hardware device, providing the lowest-level control software for that hardware. The BIOS and UEFI are firmware — they live on a chip on the motherboard, not on the hard disk. Unlike the OS, which is loaded from disk, the firmware is always present and always accessible from address 0xF0000 (in legacy BIOS) or wherever the ROM is mapped.

BIOS — Basic Input/Output System — The traditional name for x86 PC firmware, dating from the original IBM PC (1981). The BIOS was IBM's firmware implementation — it provided a standard interface (via software interrupts like INT 0x10 for video, INT 0x13 for disk) that programs could use to interact with hardware without knowing the specific hardware details. The BIOS runs in real mode, provides real-mode interrupt handlers, and has no concept of protected mode or virtual memory.

UEFI — Unified Extensible Firmware Interface — The modern replacement for BIOS, introduced in the 2000s (originally as EFI by Intel for Itanium systems). UEFI provides: a 32-bit or 64-bit execution environment (not limited to real mode), a graphical setup interface, secure boot (cryptographic verification of bootloader signatures), support for large disks (GPT partition table vs. MBR's limitations), and a standardized API. Modern x86-64 computers use UEFI, though most still support a "Legacy BIOS" or "CSM" (Compatibility Support Module) mode for backward compatibility.

CMOS — We defined this above. In the context of POST, CMOS refers specifically to the small battery-backed RAM (sometimes called NVRAM — Non-Volatile RAM) that stores BIOS/UEFI settings: boot order, system clock, RAM speed settings, CPU overclocking settings, and hardware configuration flags. POST reads these settings to know how to configure hardware.

Memory controller — The component responsible for managing all communication between the CPU and RAM chips. On older Intel systems (before 2008), the memory controller lived in a separate chip called the "north bridge" on the motherboard. On modern Intel and AMD systems, the memory controller is integrated directly into the CPU package — eliminating the latency of the external chip and significantly improving memory performance.

SPD — Serial Presence Detect — A small EEPROM (Electrically Erasable Programmable Read-Only Memory) chip on each RAM module that stores the module's specifications: capacity, data width, operating voltages, and most importantly, timing parameters. The firmware reads SPD data via a two-wire protocol called I²C (Inter-Integrated Circuit) or SMBus. Using SPD data, the firmware programs the memory controller with the correct timing parameters for the installed RAM.

RAM timing parameters — Specific delay values that the memory controller uses when communicating with RAM chips. RAM chips are not instant — they require specific times between operations. Key parameters:

  • CAS Latency (CL) — Clock cycles from sending a column address to receiving data
  • tRCD — Row to Column Delay — cycles between opening a row and accessing a column
  • tRP — Row Precharge time — cycles to close a row before opening another
  • tRAS — Row Active Strobe — minimum cycles a row must remain active

If the memory controller sends commands too fast (violating these timings), the RAM produces unreliable data. Too slow, and performance suffers unnecessarily. SPD data lets the firmware automatically set optimal timings.

Memory training — A modern RAM initialization process where the firmware runs test patterns to calibrate the physical electrical characteristics of the memory bus — signal timing, impedance matching, clock phase alignment. Modern DDR4 and DDR5 systems must train their memory connections on every boot (or every time RAM configuration changes), because the exact delay values depend on the specific physical characteristics of the PCB traces between the CPU and RAM slots.

I²C — Inter-Integrated Circuit — A simple two-wire communication protocol for low-speed communication between chips on the same circuit board. Uses two lines: SCL (clock) and SDA (data). The firmware drives the SCL clock, and devices respond on SDA. SPD EEPROMs on RAM modules are accessed via I²C/SMBus.

PCI — Peripheral Component Interconnect — A parallel bus standard for connecting peripheral devices to the motherboard. Original PCI (from the early 1990s) used a shared 32-bit bus at 33 MHz. Multiple PCI devices shared the same bus wires, communicating in turn.

PCIe — PCI Express — The modern replacement for PCI (and AGP), introduced in 2003. Unlike parallel PCI where all devices share wires, PCIe uses point-to-point serial links — each device has its own dedicated lanes directly connected to the CPU's PCIe controller. PCIe x1 is one lane (one pair of wires each direction), x4 is four lanes, x8 is eight, x16 is sixteen. Graphics cards use x16 slots for maximum bandwidth. Each PCIe lane on PCIe 5.0 provides about 4 GB/s in each direction.

Configuration space — A standardized 256-byte (PCI) or 4KB (PCIe) region of data that every PCIe device exposes, containing: Vendor ID (who made this device), Device ID (what specific device this is), Class Code (what type of device — display, storage, network, etc.), and Base Address Registers (BARs) that describe what memory ranges or I/O ports the device needs. The firmware reads these to identify and configure all installed devices.

Interrupt Vector Table (IVT) — In real mode, a 1KB region at physical addresses 0x000–0x3FF that maps interrupt numbers to handler addresses. Each entry is 4 bytes: 2 bytes of IP (offset) followed by 2 bytes of CS (segment). Entry N at address 4×N. When the CPU executes INT N in real mode, it reads the 4-byte entry at address 4×N, loads CS and IP from those bytes, and jumps to CS:IP. The firmware fills this table with pointers to its own interrupt handlers during POST.

I/O port 0x80 — A specific port address that, on the IBM PC architecture, is connected to nothing (or to a diagnostic LED header on server motherboards). Writing to port 0x80 causes a brief delay (due to the bus cycle time) and has no side effects — the written value is not stored anywhere useful. But the value can be read by an external diagnostic device called a POST card (a small board plugged into a PCIe or PCI slot that reads and displays port 0x80 values). The firmware writes a unique code to 0x80 before each POST step — if the machine freezes during POST, you can read the last code to identify which step failed.

Beep codes — Audio patterns produced by the system speaker during POST failures that occur before video is initialized. The original IBM PC used a single speaker driven by the 8254 timer chip (connected via the PPI — Programmable Peripheral Interface). Different patterns of short and long beeps (defined by the BIOS vendor — AMI, Award, Phoenix each had different codes) indicated specific failures: one long beep + three short = video error, for example. On modern systems, some motherboards have moved to LED-based POST indicators instead of beep codes.

What actually happened — from first principles

Why POST is necessary before any OS code can run

The operating system — when it eventually gets control — will assume a working computer. It assumes the CPU executes instructions correctly. It assumes RAM holds values reliably. It assumes devices are configured with non-conflicting memory ranges. It assumes the interrupt infrastructure is in place.

None of these things are true immediately after power is applied. The CPU has had no self-verification (besides the hardware state machine that initialized registers — but this does not verify that the ALU computes correctly). RAM chips are uninitialized — they contain random values, because their internal state is undefined until the memory controller sends initialization commands. Devices have been reset to their power-on default states, which often includes undefined or overlapping resource assignments.

POST creates the required environment, step by step.

The sequence of POST, in physical detail

Step 1 — CPU self-test (BIST — Built-In Self-Test)

Modern x86 CPUs support a Built-In Self-Test mode. The firmware can trigger this by placing a specific value in a register and executing a BIST instruction. The CPU runs its own internal test patterns, verifying that its caches, TLBs, and execution units function correctly. This takes 1–2ms on a modern CPU. If BIST fails, EAX contains a non-zero error code. The firmware checks this and halts if a CPU failure is detected.

Step 2 — Initialize the chipset

The CPU can read and write the chipset's configuration registers via the PCI configuration mechanism (even before full PCI initialization). The firmware programs essential chipset settings: memory controller base configuration, PCIe controller enabling, clock routing, power management settings. This step uses hard-coded values in the BIOS ROM for the specific chipset models the motherboard supports.

Step 3 — Memory controller initialization and SPD reading

The firmware programs the memory controller with initial (conservative) timing settings. It then reads SPD data from each installed RAM module via I²C. Using the SPD data, it programs the memory controller with the optimal timings for the installed RAM.

Step 4 — Memory training

On modern DDR4/DDR5 systems, the firmware runs an extensive training sequence: it sends test patterns to the RAM and measures the delay windows within which the signals are reliably received. This calibrates parameters like receiver training, write leveling, read timing. This step takes several seconds on consumer systems (which is one reason modern computers take longer to boot from a cold start than from sleep/hibernate — POST must retrain memory on every cold boot on many systems).

Step 5 — Memory test

After training, the firmware performs a basic memory test. The scope varies by firmware — a minimal test might write 0xAA then 0x55 to each location and read back; a thorough test runs multiple patterns at every address. A minimal test that just checks for obviously faulty cells takes about 1 second per GB.

After testing, the firmware knows how much RAM is installed and that it is functional. It stores the memory map (which address ranges are available and which are reserved for hardware) for later retrieval by the OS.

Step 6 — PCI/PCIe device enumeration

The firmware walks the PCIe/PCI bus. PCI devices are addressable via a special mechanism: the firmware writes a 32-bit "config address" to I/O port 0xCF8 (selecting a specific device's configuration space register) and reads/writes port 0xCFC (the "config data" register). This mechanism (called "Config Mechanism 1") is standardized across all x86 PCI systems.

The firmware reads the Vendor ID from each potential device location. Vendor ID 0xFFFF means "no device present" (the bus returns all 1s when nothing is there). Any other value means a device is present.

For each found device, the firmware:

  • Reads its Class Code to understand what type it is (0x0300 = VGA display, 0x0101 = IDE storage, 0x0200 = ethernet, etc.)
  • Reads its Base Address Registers (BARs) to find what resources it needs
  • Programs each BAR with a system-wide unique address range (no two devices can have overlapping ranges)
  • Assigns IRQ numbers (via the PIC or APIC routing table) to devices that need interrupts

Step 7 — Initialize the video subsystem

The display adapter (GPU) contains its own firmware — called the Video BIOS or VBIOS — stored in a ROM on the GPU card itself. The firmware reads the VBIOS from the GPU's ROM and executes it (this is called "Option ROM execution"). The VBIOS initializes the GPU, sets up a video mode (typically 80×25 text mode for legacy BIOS), and installs INT 0x10 (video interrupt) handlers.

After this step, the firmware can display text on screen — which is why you see the BIOS splash screen or POST output. Everything before this step has been silent.

Step 8 — Install IVT handlers

The firmware writes the Interrupt Vector Table — filling addresses 0x000–0x3FF with CS:IP pointers to real-mode interrupt service routines. These handlers implement:

  • INT 0x10: Video services (print character, set video mode, read cursor position)
  • INT 0x13: Disk services (read sectors, write sectors, check drive presence)
  • INT 0x16: Keyboard services (read keypress, check for pending keypress)
  • INT 0x15: System services (memory map query, A20 enable/disable)
  • INT 0x1A: Time/date services (read/write RTC)
  • And many others

These handlers are located in the BIOS ROM at 0xF0000. The IVT entries are just 4-byte CS:IP pointers into this region.

Step 9 — BIOS Data Area

The firmware fills the BIOS Data Area (BDA) at 0x400–0x4FF with hardware configuration information: number of serial ports and their I/O port addresses, number of parallel ports, number of floppy drives, equipment flags, keyboard state (shift key status, caps lock status), current video mode. This data is used by the IVT handlers and may be read by the bootloader.

Step 10 — POST complete signal

The firmware writes 0x00 to port 0x80 (a conventional POST code meaning "all POST tests passed") and sounds one short beep on the system speaker (the universal "all OK" signal from AMI/Award BIOSes). On modern UEFI systems, there is typically no beep by default.

What happens if POST finds a failure

Before video is initialized (very early failures):

No display is possible. The firmware can only communicate via beep codes and port 0x80 codes. It stops here — it cannot proceed because the fundamental hardware needed for any further operation is broken.

After video is initialized but before boot:

The firmware displays an error message. It may offer to enter the BIOS setup to change settings (for example, if RAM was recently changed and is not training properly, you might be prompted to reset to default timings). It may halt entirely, waiting for the user to power off and fix the hardware.

Soft errors:

Some POST failures are non-fatal. A missing optional device (like a second optical drive) is logged but does not halt POST. The firmware continues and boots normally, possibly with that device absent from subsequent configuration.

Boot Device Selection — T+~600–1500ms

The section text

"POST complete. The firmware now finds a bootable device. It checks the boot order — each device in sequence — looking for a valid boot sector signature."

Every term, explained from absolute zero

Boot order — The prioritized list of devices the firmware checks for a bootable operating system. Stored in CMOS/NVRAM, configurable in the BIOS/UEFI setup menu. Typical examples: USB first (so you can boot from a USB drive for installation or recovery), then internal SSD/HDD, then optical drive. The firmware checks each device in order, stopping at the first bootable one found.

Sector — The smallest addressable unit of storage on a disk. Historically 512 bytes per sector (established by IBM and maintained as the standard for decades). Modern drives use 4,096-byte (4KB) "Advanced Format" sectors internally but typically emulate 512-byte sectors to software for compatibility (called "512e" — 512 emulation). An NVMe SSD uses "blocks" rather than sectors but the concept is identical.

LBA — Logical Block Addressing — A way of addressing disk sectors by sequential number, starting from 0. LBA 0 is the first sector, LBA 1 is the second, and so on through all sectors on the disk. LBA replaced the older CHS (Cylinder-Head-Sector) addressing, which required knowing the physical geometry of the disk platters. All modern firmware uses LBA.

MBR — Master Boot Record — The first sector (LBA 0) of a disk, traditionally exactly 512 bytes. Contains: up to 446 bytes of bootloader code, four 16-byte partition entries describing the disk's partition layout, and the 2-byte boot signature 0x55 0xAA at the very end. The MBR is the structure the BIOS reads to determine if a disk is bootable and to begin the boot process.

0x55 0xAA — Boot signature (magic bytes) — The two bytes at offsets 510 and 511 (0-indexed) of the MBR that the firmware checks to determine if this is a valid boot sector. In memory order: byte at offset 510 = 0x55, byte at offset 511 = 0xAA. Note the apparent reversal: when stored on disk in little-endian order and read as a 16-bit value, 0xAA55 is at offset 510 (low byte 0x55 then high byte 0xAA). This is why in x86 assembly you write dw 0xAA55 — you are writing a 16-bit little-endian value whose bytes appear as 0x55 followed by 0xAA in memory order.

Partition — A defined region of a disk — a contiguous range of sectors designated for a specific purpose. A disk can be divided into multiple partitions, each formatted with its own file system. Partition information is stored in the MBR's partition table (for MBR-style disks) or in the GPT (for modern disks).

Partition table — A data structure on the disk that describes how the disk is divided into partitions. The MBR format supports up to 4 partitions, each described by a 16-byte entry. Each entry contains: a "bootable" flag byte, CHS start and end addresses, LBA start address, and sector count. The firmware reads the partition table to understand the disk's layout.

GPT — GUID Partition Table — The modern replacement for the MBR partition table, required for disks larger than 2TB (the MBR format's maximum with 32-bit LBA) and used with UEFI. GPT stores partition information in multiple redundant locations on the disk (beginning and end) and supports up to 128 partitions with 64-bit LBA addresses (supporting disks up to 9.4 ZB — zettabytes, incomprehensibly large). GPT uses GUIDs (Globally Unique Identifiers — 128-bit random numbers) to identify partition types and individual partitions.

GUID — Globally Unique Identifier — A 128-bit number used as a unique identifier. Generated using algorithms that combine timestamps, random numbers, and hardware identifiers to make collision (two different things getting the same GUID) essentially impossible. Written in a standard hyphenated hex format: xxxxxxxx-xxxx-xxxx-xxxx-xxxxxxxxxxxx.

EFI System Partition (ESP) — On UEFI systems, a special partition formatted with FAT32 that stores UEFI bootloader files. The UEFI firmware can read FAT32 directly (UEFI includes a FAT32 driver). Boot files for each OS are stored in subdirectories: Windows stores its bootloader at \EFI\Microsoft\Boot\bootmgfw.efi, Linux distributions typically use \EFI<distro>\grub.efi or similar.

FAT32 — A file system (File Allocation Table, 32-bit version) originally developed by Microsoft for Windows 95. Simple, widely supported, and chosen for the ESP because virtually all firmware can implement its reader. FAT32 uses a File Allocation Table to track which clusters belong to which files (the same concept as the FAT file system described in the program lifecycle section).

CHS — Cylinder-Head-Sector — The legacy disk addressing scheme based on physical disk geometry. A hard disk has multiple platters (cylinders), each platter has two surfaces (one per head), and each track is divided into sectors. CHS addressing specifies location as (cylinder number, head number, sector number). Limited to disks of ~504MB (with classic BIOS limits) or ~8GB (with extended BIOS), which is why LBA replaced it. Your BIOS disk read calls still use CHS in some parameters, but modern firmware translates internally to LBA.

CMOS battery — A small lithium coin cell battery (typically CR2032, 3V) on the motherboard that powers the CMOS RAM and RTC when the computer is off. Lasts 5–10 years typically. When it dies, the computer loses time synchronization (clock resets to a default date) and BIOS settings (returns to factory defaults). "CMOS battery" is a slight misnomer — the battery doesn't power the CMOS logic (which is powered by 5VSB when plugged in) but only the CMOS RAM and RTC when unplugged.

What actually happened — from first principles

How the firmware reads the disk

When the firmware needs to read a disk sector, it uses the disk controller — a chip (or integrated circuit in the CPU/chipset) that manages the physical communication with the storage device.

For a SATA hard drive or SSD, the firmware communicates with the AHCI (Advanced Host Controller Interface) controller via memory-mapped registers that were assigned during PCI enumeration. The firmware writes a command to the controller's command port: "read N sectors starting at LBA address X and place the data at memory address Y." The AHCI controller manages the actual SATA protocol communication with the drive and uses DMA (Direct Memory Access) to transfer the data directly into RAM without the CPU needing to be involved byte-by-byte.

For a USB drive, the firmware uses the XHCI (eXtensible Host Controller Interface) controller and the USB Mass Storage protocol. The USB protocol stack is more complex — multiple layers of command wrapping — but the end result is the same: sectors from the USB device appear in RAM.

For an NVMe SSD (connected directly via PCIe), the firmware uses the NVMe protocol — a much simpler and faster command set designed specifically for flash-based storage.

In all cases, after the transfer completes, a 512-byte (or more) buffer in RAM contains the data from LBA 0 of the device. The firmware checks bytes 510–511 of this buffer. If they are 0x55 and 0xAA respectively, the device is bootable.

The magic bytes — why 0x55AA

The choice of 0x55AA (stored as 0x55 then 0xAA in byte order) is partly historical convention from the original IBM PC BIOS, but it has a useful property: 0x55 = 01010101 in binary and 0xAA = 10101010 in binary. Together these two bytes use all 16 bits in an alternating pattern (0101010110101010). This alternating pattern is useful for testing data paths — if you receive 0x55AA correctly, you know that the data bus is transmitting all 16 bits correctly (no stuck-high or stuck-low bits). It was also unlikely to occur by accident at the end of an arbitrary, non-boot sector.

The 0x7C00 address — why specifically

This is a specific historical decision from the IBM PC BIOS design. The value 0x7C00 was chosen for two reasons:

First, the IBM PC had 32KB of RAM guaranteed (the minimum configuration). 32KB = 0x8000. The BIOS wanted to load the boot sector near the top of this guaranteed-present RAM while keeping the IVT (at 0x0000–0x03FF) and BDA (at 0x0400–0x04FF) intact.

Second, the boot sector itself is 512 bytes (0x200), and the BIOS wanted a stack below it. 0x7C00 - 0x200 = 0x7A00 (leave room below for stack). 0x7C00 + 0x200 = 0x7E00 (end of MBR code, beginning of free space above).

So the memory map during boot looks like:

0x0000–0x03FF : IVT
0x0400–0x04FF : BDA
0x0500–0x7BFF : Free (stack grows down from 0x7C00)
0x7C00–0x7DFF : Boot sector (512 bytes loaded here)
0x7E00–0x7FFF : Free space above boot sector
...

The stack grows downward from 0x7C00 into the free space, and additional bootloader stages can be loaded at 0x7E00 or anywhere else in free RAM.

BIOS Loads and Jumps to the MBR — T+~1500ms

The section text

"The firmware copies the 512-byte MBR to address 0x7C00 in RAM and jumps to it. Control leaves the firmware permanently. Your code runs."

Every term, explained from absolute zero

Far jump — An x86 jump instruction that changes both CS (Code Segment) and IP (Instruction Pointer) simultaneously, as opposed to a near jump which only changes IP (staying within the same code segment). A far jump to 0x0000:0x7C00 means: set CS to 0x0000 and IP to 0x7C00. The physical address of the next instruction will be (0x0000 × 16) + 0x7C00 = 0x7C00. This is how the BIOS hands control to the MBR.

Drive number — A convention used by BIOS disk services to identify which drive to access. Values: 0x00 = first floppy drive (A:), 0x01 = second floppy (B:), 0x80 = first hard disk (C: in DOS), 0x81 = second hard disk, and so on. The high bit (0x80) distinguishes hard disks from floppy drives. USB drives typically appear as 0x80 (if set up as the primary boot device, the BIOS maps them there). NVMe SSDs on UEFI systems use a completely different identification mechanism.

DL register — The low byte of the DX register (per the 8086 architecture document — DH is high byte, DL is low byte). The BIOS sets DL to the drive number before jumping to the MBR, providing the bootloader with information about which drive it was loaded from. This is necessary because the bootloader needs to read additional sectors from the same drive, and INT 0x13 needs the drive number.

ORG directive — An assembler directive (instruction to the assembler, not an actual CPU instruction) that tells the assembler: "assume that this code will be located at this address when it executes." The assembler uses this to compute the correct values for labels and memory references. ORG 0x7C00 means "all addresses in this code are relative to 0x7C00." If you have a label message: after some instructions, the assembler computes its address as 0x7C00 plus the offset of that label from the beginning of the code.

NASM — Netwide Assembler — a popular, free x86 assembler. It takes assembly language source code (human-readable mnemonics like mov ax, 0x1234) and produces binary machine code. NASM is commonly used for OS development bootloaders because it produces flat binary output (just the raw instruction bytes, no ELF or PE headers) which is exactly what the boot sector format requires. The command nasm -f bin bootloader.asm -o bootloader.bin produces a raw binary file.

Segment register loading — The process of loading a new value into a segment register (CS, DS, SS, ES). As covered in the 8086 architecture document, in real mode this is straightforward — the value loaded into DS becomes the segment base ÷ 16. Loading DS with 0x0000 means all data accesses have segment base 0 (DS:offset = 0×16 + offset = offset). Most bootloaders begin with explicit DS setup because the BIOS may have left segment registers at arbitrary values.

What actually happened — from first principles

The final BIOS operations before handoff

After finding a bootable device (checking 0x55AA at bytes 510–511 of LBA 0), the BIOS:

1 — Copies the MBR to 0x7C00:

The BIOS has already read LBA 0 into a temporary buffer (somewhere in the BIOS's own working memory, typically in the area 0x600–0x7BFF). It copies those 512 bytes to 0x7C00 using a simple memory copy loop. After this, the machine code bytes of your bootloader are sitting in RAM at addresses 0x7C00–0x7DFF.

2 — Validates (again) and sets DL:

Some BIOSes validate the signature again after copying (to confirm the copy succeeded). The BIOS sets DL = the drive number (0x80 for the first hard disk, the typical case). Other registers may be set to implementation-specific values — the BIOS specification does not guarantee the state of most registers, only DL.

3 — The far jump:

The BIOS executes something equivalent to JMP FAR 0x0000:0x7C00. This loads CS = 0x0000 and IP = 0x7C00. The CPU's next fetch-decode-execute cycle fetches from CS.base + IP = 0 + 0x7C00 = 0x7C00. The first byte at 0x7C00 is decoded as the first instruction of your bootloader.

From this moment: the BIOS has no more active role. Its code continues to sit in ROM at 0xF0000, and its IVT handlers remain at the addresses installed during POST. But the CPU is no longer executing BIOS code — it is executing your code.

What your bootloader's first instruction should do

The bootloader begins with no guarantees about most register values. It must establish its own environment before doing anything useful. The canonical first instructions of a boot sector:

asm
[BITS 16]           ; Tell NASM we're generating 16-bit code
[ORG 0x7C00]        ; Tell NASM code is loaded at 0x7C00

start:
    ; Disable interrupts while setting up segment registers
    cli

    ; Set up segment registers explicitly
    ; We cannot trust whatever values the BIOS left
    xor ax, ax       ; AX = 0
    mov ds, ax       ; DS = 0  (data segment base = 0)
    mov es, ax       ; ES = 0  (extra segment base = 0)
    mov ss, ax       ; SS = 0  (stack segment base = 0)
    mov sp, 0x7C00   ; Stack grows downward from 0x7C00
                     ; (into free space below our code)

    ; Re-enable interrupts now that segments are set up
    sti

    ; Save the drive number (BIOS set DL before jumping here)
    mov [boot_drive], dl

    ; ... proceed with loading additional sectors ...

The cli (Clear Interrupt Flag) before setting up segment registers is important: if an interrupt fired while SS was 0 but SP had not yet been set to a valid value, the interrupt handler would try to push registers onto the stack at SS:SP = 0:0 — address 0 — which would corrupt the IVT. The brief cli/sti window around segment setup prevents this.

Stage 1 Bootloader — The 510-Byte Problem

The section text

"The entire first-stage bootloader must fit in 510 bytes (512 minus the 2-byte signature). Its only job: load a larger second-stage bootloader from disk."

Every term, explained from absolute zero

INT 0x13 — BIOS Disk Interrupt — One of the software interrupt vectors installed by the BIOS during POST. Calling INT 0x13 with specific register values invokes different disk service functions. AH selects the function: AH=0x00 resets the disk, AH=0x02 reads sectors, AH=0x03 writes sectors, AH=0x42 reads via LBA extension (modern), etc.

AH register — The high byte of AX. When calling BIOS services, AH typically specifies the function number. This is a convention from the original IBM BIOS: AH selects what to do, other registers provide parameters.

Cylinder — In a hard disk drive, a cylinder is the set of all tracks at the same radial position across all platters. If a drive has 3 platters (6 surfaces) and each surface has track 42, then cylinder 42 consists of all 6 of those tracks. The term comes from the shape: if you could see all the tracks at position 42 on all platters simultaneously, they would form a cylinder.

Head — A read/write head is the component that reads from or writes to the disk surface. Each disk surface has one read/write head. A 3-platter drive has 6 heads (two per platter — one for each surface). CHS head number selects which surface (and therefore which track within the cylinder).

Sector (CHS context) — In CHS addressing, sectors are numbered within each track, starting from 1 (not 0 — a historical quirk). Each track is divided into an equal number of sectors.

INT 0x13 extensions — LBA mode — An extension to the original INT 0x13 that adds LBA-addressed disk functions. Function AH=0x41 checks if extensions are present. Function AH=0x42 reads sectors by LBA number using a "Disk Address Packet" — a small structure in memory that specifies the LBA start, sector count, and destination address. Modern bootloaders use LBA mode because CHS addressing breaks for disks larger than ~8GB.

Disk Address Packet (DAP) — A small structure in memory used with INT 0x13 extension function 0x42:

Offset 0: Size of packet (0x10 = 16 bytes)
Offset 1: Reserved (0)
Offset 2: Number of sectors to transfer (16-bit)
Offset 3: Reserved (0)
Offset 4: Destination offset (16-bit)
Offset 6: Destination segment (16-bit)
Offset 8: LBA start address (64-bit)

What actually happened — from first principles

The 510-byte constraint and what it forces

512 bytes total in the MBR. 2 bytes are the boot signature (0x55, 0xAA at offsets 510–511). The MBR partition table occupies 64 bytes at offsets 446–509 (four 16-byte entries). This leaves 446 bytes for bootstrap code in a standard MBR layout.

446 bytes. For a real OS bootloader, this is almost nothing. GRUB's Stage 1 (which lives in the MBR) uses essentially all 446 bytes just to locate and load its Stage 1.5.

For a simple OS project (like Nick Blundell's tutorial), the 512-byte MBR can contain the entire bootloader — a small amount of code that switches to protected mode and loads a tiny kernel. This is possible because:

  • The kernel being loaded is very small (a few KB)
  • The bootloader is minimal (no filesystem parsing, just raw sector reads)
  • The protected mode switch is compact in binary (the actual instructions are only ~50 bytes)

A realistic real-world bootloader (for a full OS with filesystem support) needs multiple stages:

Stage 1 (MBR, 446 bytes): Find the active (bootable) partition, locate Stage 1.5 or Stage 2 on that partition, load it into RAM, jump to it.

Stage 1.5 (~32KB, lives in the sectors between MBR and first partition): Contains filesystem drivers (so it can read Stage 2 from the filesystem). GRUB uses this layer.

Stage 2 (hundreds of KB): The full bootloader with menu, graphics, compression support, multiple OS support, module loading.

The actual disk read in assembly

A minimal bootloader reading additional sectors using INT 0x13 function 0x02 (CHS mode, for simplicity):

asm
; Read sectors using BIOS INT 0x13 function 0x02
; Assumptions: disk uses CHS, reading from the sector immediately after MBR
load_kernel:
    mov ah, 0x02         ; Function: read sectors
    mov al, 0x01         ; Number of sectors to read: 1
    mov ch, 0x00         ; Cylinder: 0
    mov cl, 0x02         ; Sector: 2 (sector 1 is the MBR; next is sector 2)
    mov dh, 0x00         ; Head: 0
    ; DL = drive number, already set from BIOS (saved earlier)
    mov dl, [boot_drive]
    ; ES:BX = destination address
    mov bx, 0x0000
    mov es, bx           ; ES = 0
    mov bx, 0x7E00       ; BX = 0x7E00 (right after MBR in RAM)
    int 0x13             ; Call BIOS disk service
    jc disk_error        ; If carry flag set, an error occurred
    ; The kernel is now loaded at 0x7E00
    jmp 0x7E00           ; Jump to it

The carry flag (CF) is set by INT 0x13 if an error occurred — a BIOS convention for returning error status. AH contains an error code if CF is set.

Stage 2 Bootloader — Setting Up for the OS

The section text

"The Stage 2 bootloader enables A20, builds a minimal GDT, switches to protected mode, and jumps to the kernel."

Every term, explained from absolute zero

A20 line — The 21st address line of the CPU (counting from A0), named A20 because it is the 21st bit (bit position 20) of the address bus, representing a value of 2²⁰ = 1,048,576 — exactly 1MB. The A20 line controls access to physical addresses at and above 1MB. The full story:

The original 8086 had only A0–A19 (20 address lines). Any address calculation that overflowed 20 bits (wrapped around past 1MB) automatically wrapped to 0 — because there was no A20 to carry the overflow. Some programs deliberately relied on this wrap-around behavior.

When the 286 added A20 (plus A21 and A22 and A23 for its 24-bit address bus), the wrap-around behavior disappeared — addresses above 1MB worked correctly, and the "overflow wraps to 0" trick broke programs that relied on it.

IBM's solution in the IBM PC AT (which used the 286): add a gate circuit that physically disconnects the A20 line, forcing it to always output 0. This restored the wrap-around behavior. The gate was controlled by a bit in the keyboard controller (the Intel 8042 chip, which IBM happened to have in their keyboard controller circuit and which had a spare output pin).

The 8042 keyboard controller's output port bit 1 became the "A20 gate" control. When bit 1 = 0: A20 is gated off (stuck at 0 — addresses above 1MB wrap to 0). When bit 1 = 1: A20 is enabled (addresses above 1MB work correctly).

Enabling A20 — Multiple methods exist because different systems implement A20 control differently:

Method 1 — BIOS INT 0x15, AX=0x2401: Ask the BIOS to enable A20. Simplest. Not all BIOSes support this.

Method 2 — Keyboard controller (8042): Send a "write output port" command to the 8042 and set bit 1 of the output port. Slow (the 8042 is slow to respond — need to wait for "ready" status). The classic method from the 286 era.

Method 3 — Fast A20 (port 0x92): A later addition where many chipsets provide a bit in I/O port 0x92 that directly controls A20 (bit 1 = A20 enable). Faster than the keyboard controller method. Not universally supported.

Method 4 — A20 is already enabled: Many modern systems (virtualized and physical) enable A20 automatically during POST and never require the bootloader to enable it. QEMU, for example, often has A20 already enabled when your bootloader starts.

Robust bootloaders try multiple methods: Enable via BIOS first, then check if A20 is actually enabled (by testing whether writing to an address above 1MB is visible — a specific test pattern), then try port 0x92, then try the keyboard controller.

DMA — Direct Memory Access — A mechanism allowing hardware devices to transfer data directly to and from RAM without the CPU being involved in each byte transfer. The disk controller reads from disk and writes directly to a specified RAM address — the CPU merely initiates the transfer and receives an interrupt when complete. This frees the CPU to do other work while the (slow) disk transfer happens.

A20 test — To verify A20 is enabled, bootloaders often use a specific test: write a distinctive value to an address just above 1MB (e.g., 0x100500). Read the address that would be aliased to it if A20 were disabled (0x000500 — same address with bit 20 cleared). If the values match (the write at 0x100500 is visible at 0x000500), A20 is still disabled — the addresses are the same due to bit 20 being gated off. If the values differ, A20 is enabled — they are genuinely different physical locations.

What actually happened — from first principles

The complete Stage 2 sequence in detail

Enabling A20:

asm
; Method: Fast A20 via port 0x92
in  al, 0x92     ; Read current value of port 0x92
or  al, 2        ; Set bit 1 (A20 enable)
and al, 0xFE     ; Clear bit 0 (reset bit — don't accidentally reset!)
out 0x92, al     ; Write back

Loading the kernel from disk (while still in real mode, using BIOS services):

The kernel binary was produced by compiling C kernel code. The bootloader knows (by convention — hardcoded address or configuration) which sectors on the disk contain the kernel. It reads those sectors into memory at 0x100000 (1MB) — this requires A20 to be enabled, because 0x100000 is the first address that requires the A20 line.

Building the GDT:

In memory (a data area declared in the bootloader's assembly source), define the GDT:

asm
gdt_start:
    ; Entry 0: Null descriptor (required)
    dd 0x00000000
    dd 0x00000000

    ; Entry 1: Code descriptor (selector = 0x08)
    ; Base=0, Limit=4GB, DPL=0, 32-bit, executable, readable
    dw 0xFFFF       ; Limit bits 0-15
    dw 0x0000       ; Base bits 0-15
    db 0x00         ; Base bits 16-23
    db 10011010b    ; Access byte: Present, DPL=0, Code, Executable, Readable
    db 11001111b    ; Flags + Limit bits 16-19: G=1(4KB), D=1(32-bit), L=0, Limit=0xF
    db 0x00         ; Base bits 24-31

    ; Entry 2: Data descriptor (selector = 0x10)
    ; Base=0, Limit=4GB, DPL=0, 32-bit, writable
    dw 0xFFFF       ; Limit bits 0-15
    dw 0x0000       ; Base bits 0-15
    db 0x00         ; Base bits 16-23
    db 10010010b    ; Access byte: Present, DPL=0, Data, Writable
    db 11001111b    ; Flags + Limit bits 16-19: G=1, D=1, L=0, Limit=0xF
    db 0x00         ; Base bits 24-31
gdt_end:

gdt_descriptor:
    dw gdt_end - gdt_start - 1    ; GDT size - 1 (the limit)
    dd gdt_start                   ; Physical address of GDT

Loading GDTR and switching:

asm
; Load the GDT
lgdt [gdt_descriptor]

; Disable interrupts (must be off during mode switch)
cli

; Set PE bit in CR0
mov eax, cr0
or  eax, 1          ; Set bit 0 (PE = Protection Enable)
mov cr0, eax

; Immediately far jump to flush prefetch queue and load
; CS with the code descriptor selector (0x08)
jmp 0x08:protected_mode_start

[BITS 32]
protected_mode_start:
    ; Now in 32-bit protected mode
    ; Load data segment registers with data descriptor (0x10)
    mov ax, 0x10
    mov ds, ax
    mov es, ax
    mov fs, ax
    mov gs, ax
    mov ss, ax
    mov esp, 0x090000   ; Set up a temporary kernel stack

    ; Jump to kernel entry point (loaded at 0x100000)
    call 0x100000

    ; If kernel returns (it shouldn't), halt
    hlt

The single most important instruction in this sequence is jmp 0x08:protected_mode_start — the far jump that simultaneously:

  1. Flushes the instruction prefetch queue (which may contain bytes fetched under 16-bit real-mode assumptions)
  2. Loads CS with selector 0x08 (pointing to our 32-bit code descriptor in the GDT)
  3. Sets CPL = RPL bits of 0x08 = 00 = Ring 0

From the [BITS 32] label onward, the assembler generates 32-bit instructions (default operand and address sizes of 32 bits, not 16). The CPU decodes them as 32-bit instructions because the D/B bit in the code descriptor (entry 0x08) is set to 1.

Kernel Entry — First Kernel Instruction — T+~1800ms

The section text

"The kernel begins. The CPU is in 32-bit protected mode. The kernel sets up its own GDT, IDT, stack, and prepares the full OS environment."

Every term, explained from absolute zero

Kernel entry point — The specific address (and therefore the specific function) that is the first code the kernel executes. In a C kernel, this is typically a function called kernel_main or kmain — the kernel's equivalent of main() in a regular C program. A small assembly stub typically calls this function after setting up the environment (stack, segment registers).

C runtime / crt0 — The C runtime is the small amount of code that runs before main() in a C program, setting up the environment that C code assumes: initializing the stack pointer, zeroing BSS (uninitialized data), copying initialized data from ROM to RAM (on embedded systems), and then calling main. In a kernel, a minimal version of this is implemented in assembly — the bootloader has set up a temporary stack, and the assembly entry point sets up a proper kernel stack and clears BSS before calling kernel_main.

PIC remapping — We covered this in the interrupt section. The 8259 PIC (Programmable Interrupt Controller) is configured by the BIOS to use interrupt vectors 0x08–0x0F for IRQ0–IRQ7. But x86 exception vectors 0x08 (Double Fault), 0x0D (General Protection Fault), and 0x0E (Page Fault) overlap these. In protected mode, a timer interrupt (IRQ0 → vector 0x08) would be indistinguishable from a double fault exception. The kernel must remap the PIC to use non-conflicting vectors (typically 0x20–0x2F for IRQ0–IRQ15).

PIC initialization sequence — Reprogramming the 8259 requires sending a specific sequence of "Initialization Command Words" (ICWs) to the PIC's command and data ports:

asm
; Remap PIC
; Master PIC: IRQ0-7 → vectors 0x20-0x27
; Slave PIC:  IRQ8-15 → vectors 0x28-0x2F

; ICW1: Begin initialization, cascade mode, ICW4 needed
mov al, 0x11
out 0x20, al    ; To master PIC command port
out 0xA0, al    ; To slave PIC command port

; ICW2: Vector offsets
mov al, 0x20    ; Master: IRQ0 → vector 0x20
out 0x21, al    ; To master PIC data port
mov al, 0x28    ; Slave:  IRQ8 → vector 0x28
out 0xA1, al    ; To slave PIC data port

; ICW3: Cascade wiring
mov al, 0x04    ; Master: slave connected to IRQ2
out 0x21, al
mov al, 0x02    ; Slave: connected to master's IRQ2
out 0xA1, al

; ICW4: 8086 mode
mov al, 0x01
out 0x21, al
out 0xA1, al

LIDT instruction — Load IDT Register — loads the 6-byte base+limit structure (same format as LGDT) into the IDTR (IDT Register). After LIDT, the CPU uses the new IDT for all exception and interrupt handling. Setting up IDT entries before LIDT is important — you do not want the CPU using an uninitialized IDT.

EOI — End of Interrupt — A signal the kernel must send to the PIC after handling each hardware interrupt. Write 0x20 to port 0x20 (master PIC command port) after handling IRQ0–IRQ7. For IRQ8–IRQ15, must also write 0x20 to port 0xA0 (slave PIC). Without EOI, the PIC will not send any more interrupts of the same or lower priority — your timer or keyboard will appear to stop after the first interrupt.

Enabling paging — Set CR0.PG (bit 31) after loading CR3 with the page directory address. CR3 must point to a valid page directory before PG is set. Setting PG=1 is similar to setting PE=1 — the next instruction fetch already goes through the MMU, so the page tables must correctly map the currently-executing code.

What actually happened — from first principles

The kernel's C entry point

After the assembly stub has set up the stack and cleared BSS, it calls the C kernel:

c
// kernel.c
void kernel_main(void) {
    // 1. Set up IDT
    setup_idt();

    // 2. Remap PIC
    pic_remap(0x20, 0x28);

    // 3. Install interrupt handlers
    idt_set_gate(0,  (uint32_t)isr0,  0x08, 0x8E);  // Division by zero
    idt_set_gate(13, (uint32_t)isr13, 0x08, 0x8E);  // GPF
    idt_set_gate(14, (uint32_t)isr14, 0x08, 0x8E);  // Page fault
    idt_set_gate(32, (uint32_t)irq0,  0x08, 0x8E);  // Timer (IRQ0 → vector 0x20)
    idt_set_gate(33, (uint32_t)irq1,  0x08, 0x8E);  // Keyboard (IRQ1 → vector 0x21)

    // 4. Load IDT
    idt_flush((uint32_t)&idt_ptr);

    // 5. Enable interrupts
    asm volatile("sti");

    // 6. Now the kernel is running with full interrupt support
    // Print something, or start the scheduler, etc.
    terminal_writestring("Kernel loaded. Interrupts enabled.\n");

    // 7. Idle loop — keep the CPU busy until interrupted
    for(;;) {
        asm volatile("hlt");   // Pause until next interrupt
    }
}

The hlt instruction in the idle loop is important: it tells the CPU to enter a low-power state and wait for the next interrupt. When an interrupt arrives, the CPU wakes, handles it (runs the ISR), and then the hlt instruction is re-executed. Without hlt, the idle loop would spin at full CPU speed consuming 100% CPU time doing nothing — burning power and generating heat.

First User Process — Ring 0 to Ring 3 — T+~1800–2000ms

The section text

"Kernel subsystems ready. The kernel creates the first user process — init on Linux — dropping from Ring 0 to Ring 3 for the first time."

Every term, explained from absolute zero

PID — Process Identifier — A unique number assigned to each running process by the operating system. PIDs are sequential integers starting from 1 (some OSes reserve PID 0 for the idle process or the kernel itself). PID 1 is always init (or systemd) on Linux — the first user-space process. All other processes are descendants of PID 1.

init — The first user-space process on Unix/Linux systems. Named "init" for "initialization." Its job: read the system configuration files (/etc/inittab or systemd unit files), start all system services (networking, logging, login manager, etc.), and become the "parent of all processes." If init dies, the kernel panics — the entire process hierarchy has lost its root.

systemd — A modern init replacement used by most Linux distributions. Unlike traditional init (which ran shell scripts sequentially), systemd manages services as "units," starts them in parallel where possible, tracks their dependencies, and provides extensive logging and process management.

smss.exe — Session Manager Subsystem — Windows's equivalent of init. The first user-mode process after the kernel initializes. It starts the Windows subsystem (csrss.exe), the service control manager (services.exe), and the login process (winlogon.exe).

IRET to user mode — The mechanism for transitioning from Ring 0 to Ring 3 for the first time (and for returning from system calls/interrupts). IRET pops SS, ESP, EFLAGS, CS, and EIP from the kernel stack in that order (for a ring transition). If the popped CS has RPL=3, the CPU drops to Ring 3. This is covered completely in the privilege levels section.

PCB — Process Control Block — The kernel's data structure for tracking one process. Contains: the process's PID, state (running/waiting/sleeping/zombie), saved register values (for context switching), virtual address space information (CR3 value / page table root), open file descriptors, signal handlers, priority, accounting data (CPU time used), and pointers to parent and child processes.

What actually happened — from first principles

The complete sequence to launch the first user process

At this point, the kernel has: a working IDT, working interrupt handlers, a working timer, possibly a working keyboard handler, and some form of memory management. It now creates the first user process:

1 — Allocate physical frames for the process:

The kernel's physical memory allocator (a bitmap or free-list tracking which physical frames are in use) allocates frames for:

  • The kernel binary (already in RAM, mapped in this process's kernel address space)
  • The process's code (read from disk — the /sbin/init executable)
  • The process's initial stack (one frame = 4KB)
  • The process's page directory and initial page tables

2 — Build a page directory for this process:

The kernel creates a new page directory. It maps:

  • Kernel virtual address range (0xC0000000–0xFFFFFFFF on Linux) → the kernel's physical pages. These pages have the User/Supervisor bit = 0 (supervisor only — user code cannot access them).
  • User virtual address range (0x00000000–0xBFFFFFFF): maps the process's code, data, and stack to the allocated physical frames, with User/Supervisor bit = 1 (user accessible).

3 — Set up the TSS (Task State Segment):

The TSS's SS0 and ESP0 fields are set to the kernel's stack segment selector and the address of this process's kernel stack. When this process makes a system call or takes an interrupt while in Ring 3, the CPU uses SS0:ESP0 from the TSS as the Ring 0 stack.

4 — Prepare the transition:

The kernel switches to the new process's page directory (writes CR3 with the new page directory's physical address). Then sets up the kernel stack for the IRET:

asm
; Push what IRET will pop, in reverse order
; (IRET pops: EIP, CS, EFLAGS, ESP, SS)
push user_ss       ; SS for Ring 3 (data descriptor with RPL=3, e.g., 0x23)
push user_esp      ; Initial stack pointer in user space
push eflags        ; EFLAGS with IF=1 (interrupts enabled in user mode)
push user_cs       ; CS for Ring 3 (code descriptor with RPL=3, e.g., 0x1B)
push user_eip      ; Entry point of the process (from the executable's header)
iret               ; Pop all five, switch to Ring 3, jump to user_eip

5 — The IRET executes:

EIP is loaded with the process's entry point (0x08048000 on classic Linux x86, or wherever the executable was mapped). CS is loaded with a selector having RPL=3 — CPL instantly becomes 3. The CPU is now executing user code. EFLAGS is restored with IF=1 — hardware interrupts are enabled in user mode, which is necessary for the timer to work.

6 — The system is up:

The first user process is running. When it makes its first system call (to open a file, read the configuration, etc.), INT 0x80 (or SYSCALL) fires, the CPU transitions to Ring 0, the system call handler runs, and returns to Ring 3. The timer fires 100 times per second, the kernel's timer ISR runs (in Ring 0), decides to continue running this process (since it's the only one), and returns. Eventually, init starts forking child processes for system services, each fork creating a new PCB, a new page directory, and eventually more IRET-to-Ring-3 transitions.

The boot process is complete.
