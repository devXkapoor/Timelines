Phase 01 — Hardware Initialization
T + 0 ms
Power Button Pressed — The Signal Chain
Voltage reaches the motherboard. The Power Supply Unit negotiates with the chipset. Nothing runs until the PSU says it is ready.
Power Button Pressed — The Signal Chain
[close]

When you press the power button, you are closing a physical switch on the motherboard. This switch connects a specific motherboard pin — called PWR_BTN# (the # means "active low" — the signal is active when the pin is at low voltage) — to ground, pulling it low.

The motherboard's chipset (a dedicated chip that manages communication between the CPU, RAM, storage, and other components) detects this signal and sends a PS_ON# signal to the PSU (Power Supply Unit) — the box inside your computer that converts AC mains power (from your wall outlet) to the DC voltages the computer needs.

The PSU begins powering up its internal circuits. It does NOT immediately send full power to the motherboard — it first ensures all its output voltages are stable and within specification. When ready, it asserts a signal called PWR_OK (Power Good) — a single wire going from the PSU back to the motherboard carrying a high voltage meaning "my outputs are stable, it is safe to proceed."

The time between pressing the button and PWR_OK going high: typically 100–500 milliseconds. During this time, the CPU is held in a reset state — a special condition where it is powered but all its internal registers are in defined initial states and it executes nothing. The reset signal is held by the chipset until PWR_OK arrives.

DC Voltages the PSU provides +12V — Drives CPU power regulators, PCIe slots, drive motors
+5V — Older logic, USB ports, some storage
+3.3V — Modern logic chips, RAM, some CPU rails
-12V — Legacy serial ports
+5VSB — Standby rail, always on while plugged in (powers the power button itself)
→ OS relevance The PSU's PWR_OK signal is the physical starting gun. Without it, the CPU never leaves reset. This is why an unstable PSU — one that asserts PWR_OK before its voltages are actually stable — causes mysterious crashes and memory corruption that look like software bugs but are actually hardware failures.
T + ~100–500 ms
CPU Reset State — All Registers to Known Values
PWR_OK arrives. The chipset releases the CPU's reset pin. The CPU's internal state snaps to fixed, defined values. It is ready to fetch its first instruction.
CPU Reset State — All Registers to Known Values
[close]

When the chipset releases the CPU's RESET# pin, the CPU's internal hardware forces every register to a specific, documented initial value. This is not software — it is transistor-level logic inside the CPU that fires when RESET# transitions from low to high. The values are specified in Intel's documentation and have been constant across every x86 CPU for 40 years.

The initial register values after reset on an x86 CPU:

x86 CPU register values after RESET EIP (Instruction Pointer): 0x0000FFF0
CS (Code Segment): selector=0xF000, base=0xFFFF0000, limit=0xFFFF
DS, ES, SS, FS, GS: selector=0x0000, base=0x00000000, limit=0xFFFF
EFLAGS: 0x00000002 (only bit 1 set — a reserved bit always 1)
CR0: 0x60000010 (PE=0 — real mode; PG=0 — paging off)
All general registers (EAX, EBX, ...): 0x00000000

The critical entry: CS base = 0xFFFF0000 and EIP = 0x0000FFF0. The first instruction address is CS.base + EIP = 0xFFFF0000 + 0xFFF0 = 0xFFFFFFF0. This is the reset vector — address 0xFFFFFFF0 in physical memory, 16 bytes below the top of the 4GB address space.

Note that CR0's PE bit is 0 — the CPU starts in real mode, the 8086-compatible mode. Every x86 CPU ever made, from the original 8086 to a modern Core i9, boots in real mode. This backward compatibility has been maintained for 46 years.

→ OS relevance This is why your OS bootloader begins in real mode — not by choice but because every x86 CPU mandatorily starts there. Your OS project's bootloader, loaded at 0x7C00, runs in real mode and must explicitly perform the protected mode switch. The CPU does not help you — it starts at its mandated reset vector and runs whatever code is there.
Phase 02 — Firmware (BIOS / UEFI)
T + ~500 ms · First instruction
The Reset Vector — First Instruction Executed
The CPU fetches from 0xFFFFFFF0. This address is mapped to the firmware ROM chip, not RAM. The firmware's first instruction is here — a jump to its main body.
The Reset Vector — First Instruction Executed
[close]

The CPU puts address 0xFFFFFFF0 on the address bus. But RAM has not been initialized yet — the RAM chips contain random, meaningless data. So how does the CPU find valid code to execute?

The answer: the motherboard's address decoder (logic in the chipset that decides which hardware component responds to which address range) is configured so that accesses to the very top of the address space (the region around 0xFFFFFFF0) are not routed to RAM chips. Instead, they are routed to the firmware ROM chip — a non-volatile storage chip on the motherboard that contains the BIOS or UEFI firmware code.

This ROM chip responds to the address bus exactly as RAM would — the CPU cannot tell the difference from an electrical perspective. The ROM's decoder activates the appropriate byte-group and puts the firmware's bytes onto the data bus. The CPU receives them as its first instruction.

What is at 0xFFFFFFF0? Almost always: a far jump instruction — JMP FAR — to the actual start of the BIOS/UEFI code body, which lives somewhere else in the ROM (typically around 0xF0000 in the legacy 1MB memory map, the region called "BIOS ROM" in the IBM PC memory layout). This jump is necessary because only 16 bytes exist between 0xFFFFFFF0 and the top of the 4GB address space at 0xFFFFFFFF — not enough room for the full firmware.

Memory Map at Reset (legacy BIOS layout) 0x00000000 – 0x000003FF : Interrupt Vector Table (IVT) — 1KB
0x00000400 – 0x000004FF : BIOS Data Area (BDA) — 256 bytes
0x00000500 – 0x0009FFFF : Free conventional RAM
0x000A0000 – 0x000BFFFF : Video RAM (VGA) — 128KB
0x000C0000 – 0x000EFFFF : Expansion ROM space — 192KB
0x000F0000 – 0x000FFFFF : BIOS ROM shadow — 64KB
0xFFFFFFF0 : Reset vector (ROM, not RAM)
→ OS relevance The BIOS ROM at 0xF0000 is still visible in real mode as segment 0xF000. When your bootloader calls BIOS services (INT 0x10 for video, INT 0x13 for disk), it is calling into code that lives in this region. This is why BIOS services only work in real mode — the BIOS code was written for the real-mode address model.
T + ~500–600 ms
POST — Power-On Self-Test
The firmware tests the CPU, counts and tests RAM, identifies installed hardware, and initializes everything that must work before the OS loads. If something fails, POST stops and reports an error.
POST — Power-On Self-Test
[close]

POST (Power-On Self-Test) is the firmware's hardware verification and initialization sequence. It runs before any OS code is loaded. The firmware is in complete control of the machine during POST.

CPU verification: The firmware executes a sequence of self-test instructions on the CPU, checking that basic arithmetic, register operations, and flag behavior are correct. If the CPU fails, nothing can proceed.

RAM initialization and testing: RAM chips require initialization — sending specific command sequences via the memory controller to configure their operating parameters (timing, voltage, bus width). The firmware negotiates these parameters via a protocol called SPD (Serial Presence Detect) — reading a small EEPROM chip on each RAM module that stores its timing specifications. After initialization, the firmware performs a basic memory test — writing known patterns to each address and reading them back to detect faulty cells.

Hardware enumeration: The firmware walks the PCI/PCIe bus — the high-speed interconnect that connects the CPU to graphics cards, network cards, storage controllers, and other expansion devices. Each device has a configuration space containing its device ID, vendor ID, and resource requirements. The firmware reads these, assigns memory ranges and IRQ numbers to each device, and initializes basic device operation.

Setting up the IVT: The firmware installs real-mode interrupt handlers in the Interrupt Vector Table at address 0x0000–0x03FF. Each of the 256 4-byte IVT entries is a CS:IP pointer to an interrupt service routine in the BIOS ROM. These handlers implement BIOS services (disk I/O, video output, keyboard input) that the bootloader will use.

POST error codes: During POST, the firmware continuously writes diagnostic codes to I/O port 0x80 (a special port with no side effects, readable by external diagnostic cards). If POST fails before video is initialized, the BIOS communicates errors via beep codes — patterns of short and long beeps from the motherboard speaker with specific meanings defined by each BIOS vendor.

→ OS relevance The IVT installed by POST is what makes BIOS services available to your bootloader. INT 0x10 (video), INT 0x13 (disk), INT 0x16 (keyboard) — these all work because POST installed their handlers. Once your OS switches to protected mode, the IVT is no longer used (protected mode uses the IDT instead), and BIOS services become inaccessible.
T + ~600–1500 ms
Boot Device Selection — Where Is the OS?
POST complete. The firmware now finds a bootable device. It checks the boot order — each device in sequence — looking for a valid boot sector signature.
Boot Device Selection — Where Is the OS?
[close]

POST complete. The firmware now searches for a bootable device according to the boot order — a list stored in CMOS (a small battery-backed RAM chip on the motherboard that stores BIOS settings across power cycles). The boot order might be: USB first, then hard disk, then DVD, then network.

For each device in order, the firmware tries to find a valid boot sector:

For a hard disk or USB drive: The firmware reads the very first sector — sector 0, LBA address 0 — which is 512 bytes (traditional sector size). This sector is called the MBR (Master Boot Record). The firmware checks bytes 510 and 511 of this sector: if they contain the value 0x55 0xAA (the "boot signature" or "magic bytes"), this device is considered bootable. If the signature is absent, the firmware moves to the next device in the boot order.

The 0x55AA signature is a sanity check, not true security. Any sector with these two bytes at the end will be treated as a valid boot sector, regardless of what the preceding 510 bytes contain. This is how your bootloader announces itself to the firmware.

Alternative: UEFI boot entries. Modern systems using UEFI firmware (the modern replacement for BIOS) work differently — instead of looking for a fixed-position signature, UEFI reads a dedicated EFI System Partition (ESP) formatted with the FAT32 file system and looks for bootloader executable files (with a .EFI extension) in specific paths. UEFI provides a much richer environment than legacy BIOS but operates on the same fundamental principle: firmware finds and hands control to a bootloader.

MBR Layout (512 bytes) Bytes 0–445 : Bootstrap code (the first-stage bootloader)
Bytes 446–461 : Partition entry 1 (16 bytes)
Bytes 462–477 : Partition entry 2 (16 bytes)
Bytes 478–493 : Partition entry 3 (16 bytes)
Bytes 494–509 : Partition entry 4 (16 bytes)
Bytes 510–511 : Boot signature: 0x55, 0xAA
→ OS relevance In your OS project, the file you write that gets assembled into a 512-byte binary and placed at the first sector of a disk image IS the MBR. The 0x55AA bytes at the end — which you specify with dw 0xAA55 (little-endian) in NASM — are the only reason the BIOS considers your disk bootable. Without these two bytes in exactly the right position, the firmware silently skips your disk.
Phase 03 — The Bootloader
T + ~1500 ms · Handoff to bootloader
BIOS Loads and Jumps to the MBR
The firmware copies the 512-byte MBR to address 0x7C00 in RAM and jumps to it. Control leaves the firmware permanently. Your code runs.
BIOS Loads and Jumps to the MBR
[close]

The firmware has found a device with the 0x55AA signature. It now performs the handoff:

Step 1 — Copy: The firmware uses its own disk I/O routines to read the 512-byte MBR from the device and copy it into RAM at physical address 0x7C00. This address is a historical convention established by IBM in 1981 for the original IBM PC. The value 0x7C00 = 32KB - 1KB = 31,744 bytes. IBM chose this address to place the boot sector near the top of the lowest 32KB of RAM, leaving the IVT (0x000–0x3FF) and BDA (0x400–0x4FF) intact below it.

Step 2 — Register setup: Before jumping, the firmware sets certain register values that the bootloader can use: DL (the lower byte of DX) receives the drive number of the boot device (0x00 = first floppy, 0x80 = first hard disk, 0x81 = second hard disk, etc.). The bootloader needs this to know which device to read further sectors from.

Step 3 — Far jump: The firmware executes a far jump to 0x0000:0x7C00 (segment 0, offset 0x7C00). CS is set to 0x0000, IP to 0x7C00. The CPU is now executing the first byte of your MBR code.

From this moment, the firmware is no longer in control. It has surrendered the CPU entirely. The BIOS code remains in memory (the ROM is still mapped at 0xF0000) and its IVT entries still point to working interrupt handlers — so INT instructions that call BIOS services still work. But the CPU's fetch-decode-execute cycle is now running your bootloader's instructions.

CPU state when bootloader begins CS = 0x0000, IP = 0x7C00 (first MBR byte)
DL = boot drive number (0x80 for first HDD)
Mode = real mode (16-bit)
IVT at 0x0000: BIOS interrupt handlers installed
RAM: mostly uninitialized (only MBR at 0x7C00 is valid)
→ OS relevance This is the entry point of your OS project. The first instruction in your bootloader runs at 0x7C00. Your first lines of assembly set up the segment registers (since BIOS might leave them in various states), set up a stack, and begin your bootloader's work. The org 0x7C00 directive in NASM tells the assembler to compute all address offsets relative to this starting address.
T + ~1500 ms · Running in MBR
Stage 1 Bootloader — The 510-Byte Problem
The entire first-stage bootloader must fit in 510 bytes (512 minus the 2-byte signature). Its only job: load a larger second-stage bootloader from disk.
Stage 1 Bootloader — The 510-Byte Problem
[close]

510 bytes is a severely constrained environment. The firmware loaded exactly one sector. Your code occupies at most 510 bytes (bytes 0–509 of the MBR; bytes 510–511 are the 0x55AA signature). For a simple OS, this may be sufficient — your kernel and bootloader logic could fit within this space. For a real OS, it is not.

In a multi-stage boot, the Stage 1 bootloader's single purpose is to load a larger Stage 2 bootloader from disk. To do this from real mode using BIOS services:

Using INT 0x13 (BIOS disk service): The bootloader populates specific registers with the disk read parameters and executes INT 0x13, function 0x02 (read sectors):

INT 0x13 / AH=0x02 — Read Disk Sectors AH = 0x02 (function: read sectors)
AL = number of sectors to read
CH = cylinder number (low 8 bits)
CL = sector number (bits 0–5) + cylinder high (bits 6–7)
DH = head number
DL = drive number (from register at boot time)
ES:BX = destination address in RAM

This "CHS" (Cylinder-Head-Sector) addressing is the legacy way to specify disk locations. Modern bootloaders use INT 0x13 extension function 0x42 with "LBA" (Logical Block Address) — a simpler flat sector number — but the principle is identical: ask the BIOS to read N sectors from a specific disk location into a specific RAM address.

The Stage 1 bootloader reads the Stage 2 bootloader (which might be several KB to several hundred KB) into a known address in RAM — commonly 0x0500 (just above the BDA) or 0x1000 — and then jumps to it.

→ OS relevance In Nick Blundell's OS book, the bootloader is simple enough to fit in one sector, and the "kernel" is small enough to be loaded directly by that bootloader. In production OS bootloaders (GRUB, for instance), the MBR Stage 1 is 446 bytes of code that loads GRUB's Stage 1.5 (embedded after the MBR in the gap before the first partition), which loads Stage 2 (the full GRUB menu and module system) from the filesystem.
T + ~1500–1800 ms
Stage 2 Bootloader — Setting Up for the OS
Now with more space. The Stage 2 bootloader loads the OS kernel from disk, switches the CPU to 32-bit protected mode, and jumps to the kernel. This is where your OS project lives.
Stage 2 Bootloader — Setting Up for the OS
[close]

The Stage 2 bootloader has more space and more responsibilities. Its complete sequence:

1 — Enable the A20 line: A hardware quirk from the IBM PC era. The original 8086 had only 20 address lines (A0–A19). When a segment:offset address exceeded 1MB (0xFFFFF + 1 = 0x100000), it wrapped around to 0x00000. Programs relied on this wrapping behavior. The 286 introduced a 24th address line (A20), which broke the wrap-around. To maintain backward compatibility, IBM added a circuit to physically disable A20 by default. Your bootloader must explicitly enable A20 (through a keyboard controller register, a BIOS call, or a FAST A20 chipset register) so that physical addresses above 1MB are accessible.

2 — Load the kernel from disk: The bootloader reads the kernel binary from the disk (using INT 0x13 while still in real mode) and places it at a target address in RAM — commonly 0x100000 (1MB) or higher, since addresses below 1MB are occupied by the IVT, BDA, video memory, and BIOS ROM. This requires A20 to already be enabled.

3 — Build the GDT: In memory (not on disk), the bootloader constructs the Global Descriptor Table — the array of 8-byte descriptors that defines memory segments for protected mode. At minimum: a null descriptor (entry 0, required by x86 spec), a 32-bit code descriptor (base=0, limit=4GB, DPL=0, executable), a 32-bit data descriptor (base=0, limit=4GB, DPL=0, writable). Then executes LGDT to load the GDTR register.

4 — Switch to protected mode: Set PE (bit 0) of CR0 to 1. Immediately execute a far jump to flush the prefetch queue and reload CS with a selector pointing to the 32-bit code descriptor. Now the CPU is in 32-bit protected mode.

5 — Set up 32-bit segment registers: Load DS, ES, FS, GS, SS with the data descriptor selector (typically 0x10). Set up the stack (ESP to a valid address in RAM).

6 — Jump to the kernel: Far jump (or call) to the kernel's entry point address in RAM. The bootloader is done.

Typical memory layout after bootloader completes 0x00000000 – 0x000003FF : IVT (no longer used in protected mode)
0x00000400 – 0x000004FF : BIOS Data Area
0x00007C00 – 0x00007DFF : Stage 1 bootloader (512 bytes)
0x00008000 – 0x0000FFFF : Stage 2 bootloader / stack area
0x00100000 – ... : Kernel binary loaded here
0xFFFF0000 – 0xFFFFFFFF : BIOS ROM (still mapped)

Phase 04 — OS Kernel Takes Control
T + ~1800 ms · Kernel entry
Kernel Entry Point — First Kernel Instruction
The kernel begins. The CPU is in 32-bit protected mode. The kernel sets up its own GDT, IDT, stack, and prepares the full OS environment before any user process can run.
Kernel Entry Point — First Kernel Instruction
[close]

The bootloader's far jump lands at the kernel's entry point — the first instruction of the kernel binary. The kernel is now in control. It runs at CPL=0 (Ring 0), using the flat 32-bit segments the bootloader set up.

The kernel's first task is to establish its own environment, more carefully than the bootloader's minimal setup. The typical sequence:

1 — Set up a proper kernel stack: The bootloader may have set up a temporary stack. The kernel establishes its own stack at a permanent, well-known location — large enough for kernel operations and interrupt handling.

2 — Remap the PIC: As covered in the interrupt document, the BIOS configured the 8259 PIC to use interrupt vectors 0x08–0x0F for IRQ0–IRQ7. These overlap with CPU exception vectors (double fault at 0x08, GPF at 0x0D). The kernel must remap IRQ0–IRQ7 to vectors 0x20–0x27 and IRQ8–IRQ15 to 0x28–0x2F to eliminate this conflict.

3 — Build and load a proper IDT: The bootloader's GDT was minimal. The kernel now builds a full IDT — handlers for all CPU exceptions (vectors 0–31), handlers for all remapped IRQs (0x20–0x2F), and a system call gate (0x80 or equivalent). Loads the IDTR via LIDT.

4 — Enable interrupts: Execute STI (Set Interrupt Flag) to enable hardware interrupts. From this moment, hardware events (timer, keyboard) will cause the CPU to jump to the kernel's ISRs.

5 — Detect memory: Query how much RAM the machine has — typically via information the bootloader gathered using BIOS interrupt 0x15 (e15h function E820, which returns a memory map) before switching to protected mode, and passed to the kernel in a known location.

6 — Initialize paging: Build page tables (page directory + page tables), set CR3, set CR0.PG=1. From this moment, all addresses are virtual and go through the MMU.

7 — Initialize kernel subsystems: Memory allocator (for dynamic kernel memory), scheduler data structures, device driver initialization, file system mounting, network stack.

→ OS relevance Steps 1–4 are the scope of a basic OS project. Steps 5–7 are the continuation. Each step is now fully explained by the documents we have built. T + ~1800–2000 ms
First User Process — The OS Hands Control to a Program
Kernel subsystems ready. The kernel creates the first user process — init on Linux, Session Manager on Windows — dropping from Ring 0 to Ring 3 for the first time.
First User Process — Ring 0 to Ring 3 for the First Time
[close]

Kernel initialization complete. Everything needed to run user programs is in place. The OS now creates and launches its first user-space process.

On Linux: The kernel attempts to execute /sbin/init (or /init, or systemd) — the first user-space process, always assigned PID 1. All other processes on the system are descendants of PID 1. Init is responsible for starting all system services (networking, login managers, etc.).

On Windows: The first user-space process is smss.exe (Session Manager Subsystem), which launches the Windows subsystem, the service control manager, and eventually the login screen (winlogon.exe).

The Ring 0 → Ring 3 transition: To run a user process, the kernel: allocates physical frames for the process's code, data, and stack; builds a page directory and page tables for the process with appropriate user/supervisor bits; sets CR3 to the new page directory; pushes the user-mode CS:EIP (code segment with RPL=3 and entry point) and SS:ESP (stack with RPL=3) onto the kernel stack; executes IRET. The IRET pops CS with RPL=3, which sets CPL=3, and the CPU is now executing user code in Ring 3 for the first time.

From this moment, the system is "up." Any interaction with hardware (keyboard, disk, network, screen) goes through the kernel via system calls — the only legal crossing point from Ring 3 to Ring 0 that the hardware enforces.

→ OS relevance The IRET that first drops to Ring 3 is the culmination of the entire CPU-architecture story we have built. The privilege levels, the CPL in CS, the TSS providing the kernel stack, the page tables with user/supervisor bits — all of it converges in that single IRET instruction. Every subsequent user process creation follows the same pattern, scaled up with process management infrastructure.
