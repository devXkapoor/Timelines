Era V · The Classic Algorithms

Divide, Conquer, Sort, Search

Recursion stops being a curiosity and becomes a deliberate design strategy — and a handful of algorithms invented in the 1960s remain, today, the most-implemented pieces of code in the history of computing.

The shift in this era. Era IV gave us a toolbox of data structures and a precise language (Big-O) for comparing them. This era asks: given that toolbox and that language, what are the actual algorithms — the step-by-step procedures — that people discovered for some of the most common tasks imaginable? Two tasks dominate this era because they underlie an enormous fraction of all computing: sorting (putting things in order) and searching (finding something).

This era also crystallizes a design pattern that you've seen the shadow of throughout this entire timeline — Euclid's GCD shrinking to a smaller GCD problem, the binary search tree eliminating half the possibilities with one comparison, the Egyptian doubling trick — into an explicit, named, reusable strategy: divide-and-conquer. Once this pattern is named and understood as a general technique (rather than something that just happens to work in a few specific cases), it becomes a tool you can deliberately reach for when designing a new algorithm — and a huge fraction of "clever" algorithm ideas, even decades later, are applications of this one pattern.

Pre-1960s, formalized 1960s · The "obvious" approach

Insertion Sort and the O(n²) Baseline

+
The Source
Insertion sort builds a sorted sequence one element at a time: for each new element, it is inserted into its correct position among the previously-sorted elements, shifting larger elements over as needed — the same method many people use intuitively when sorting a hand of playing cards.
Vocabulary
Sortingarranging a collection of items into a specified order (e.g., numbers from smallest to largest, words alphabetically).
In-placean algorithm that rearranges data within its original storage, without needing a significant amount of extra memory to hold a separate copy.
Comparison-baseda sorting algorithm that works only by comparing pairs of elements ("is A bigger than B?") and rearranging based on the answer — as opposed to algorithms that exploit special knowledge about the data (covered in a later entry).
Stable (sort)a sorting algorithm is "stable" if, when two elements are considered equal, their original relative order is preserved in the output. This matters when sorting complex records by one field but wanting ties to retain their prior order.
First Principles

It's worth starting here, with the "obvious" approach, for an important reason: understanding why the obvious approach is limited is what makes the cleverer approaches (next entries) feel like genuine insights rather than arbitrary tricks.

Insertion sort works exactly the way most people sort a hand of playing cards: keep a sorted pile in your hand, and each time you draw a new card, slide it into the correct position among the cards you're already holding — shifting other cards over as needed.

Each new element may need to shift past several already-sorted elements to find its place — inserting 1 had to pass 2, 5, and 8.

Why is this O(n²)? Consider the worst case: an array sorted in reverse order. Every single new element is smaller than everything already placed, so every new element must shift past all previously-placed elements. The first element requires 0 shifts, the second requires up to 1, the third up to 2, ... the n-th requires up to n−1. Adding these up: 0+1+2+...+(n−1), which works out to roughly n²/2 — and Big-O drops the constant, giving O(n²).

This O(n²) pattern — "for each of n items, do up to n units of work" — is the single most common source of "slow" algorithms, and recognizing it is a basic skill: any time you have a loop inside a loop, where both loops can run up to n times, you very likely have an O(n²) algorithm. For small n (say, n ≤ 1,000), O(n²) is perfectly fine — a million operations is instant on modern hardware. The problem appears at scale: n = 100,000 gives 10 billion operations, which is too slow for typical time limits. This is exactly the gap that motivates the next two entries.

Why insertion sort still matters Despite being O(n²), insertion sort has a genuinely useful property: it's extremely fast on data that's already mostly sorted (in that case, each new element needs very few shifts) — and it's simple enough that, for very small n (roughly under 10–20 elements), its low overhead actually makes it faster in practice than more "sophisticated" algorithms. Several real-world sorting implementations use insertion sort as a final step for small sub-arrays within a larger, more complex algorithm — a pattern you'll recognize in the next entry.
1945, popularized 1960s · John von Neumann (described it), widely adopted

Merge Sort — Divide-and-Conquer Gets a Name

+
The Source
Merge sort works by splitting a sequence in half, recursively sorting each half, and then merging the two sorted halves back together into a single sorted sequence — a "merge" step that can be done efficiently because both halves are already sorted.
Vocabulary
Divide-and-conquera strategy for solving a problem by (1) dividing it into smaller subproblems of the same type, (2) solving each subproblem (often by applying the same strategy recursively), and (3) combining the subproblems' solutions into a solution for the original problem.
Base casethe simplest version of a problem, small enough to solve directly without further recursion — for sorting, a sequence of length 0 or 1 is already sorted, with nothing to do.
Merge (step)combining two already-sorted sequences into one sorted sequence, by repeatedly comparing the front elements of each and taking the smaller one.
Recursion treea way of visualizing recursive calls as a tree, where each node represents one call, and its children represent the recursive calls it makes. The depth of this tree, and the work done at each level, together determine the algorithm's overall complexity.
Auxiliary spaceextra memory an algorithm needs beyond the input itself. Merge sort, in its standard form, needs O(n) auxiliary space to hold the merged results — unlike insertion sort, which is in-place.
First Principles

Here is the divide-and-conquer pattern, explicitly, for the first time in this timeline: to sort an array, (1) divide it into two halves, (2) recursively sort each half (i.e., apply this exact same procedure to each half — "the same kind of problem, but smaller," exactly like Euclid's GCD reducing to a smaller GCD), and (3) combine the two sorted halves via the merge step.

Splitting continues until pieces are size 0 or 1 (the base case). All the "real work" happens during the merges, which combine already-sorted pieces.

Now, the complexity. Splitting in half repeatedly: how many times can you halve n before reaching pieces of size 1? That's exactly log₂(n) — the same quantity that appeared in the binary search tree's height. So there are roughly log(n) "levels" of splitting. At each level, the total work done across all the merges at that level is roughly n (every element gets touched once during a merge at each level, because merging is "walk through both halves once"). Total work: roughly n × log(n) — giving merge sort its famous O(n log n) complexity.

Go back to the numbers from Era IV: for n = 1,000,000, O(n²) is a trillion operations (too slow), but O(n log n) is about 1,000,000 × 20 = 20 million operations (fast, well under a second). Merge sort isn't a minor improvement over insertion sort — it's a fundamentally different category, and this category difference is exactly what divide-and-conquer, done right, can achieve: turning an O(n²) problem into an O(n log n) one by recursively halving.

The "recursion tree" framing is worth sitting with, because it's the standard way divide-and-conquer algorithms are analyzed from here on: draw a tree where the root is "sort all n elements," its two children are "sort the left n/2" and "sort the right n/2," their children are "sort n/4" each, and so on, down to leaves of size 1. The tree has log(n) levels (since you halve at each level until reaching 1), and the total work at each level is proportional to n (because the merge work across all pieces at one level adds up to roughly n regardless of how many pieces there are). log(n) levels × O(n) work per level = O(n log n) total. This "count the levels, count the work per level, multiply" technique is how you'll analyze most divide-and-conquer algorithms you encounter.

Why this matters for CP Merge sort itself is rarely hand-written in contests (most languages have a fast built-in sort), but the technique — and especially the merge step — reappears constantly: merging is the core of algorithms for counting inversions, certain range-query problems, and "merge sort tree" structures (an advanced data structure combining segment trees, which you'll meet later, with merge sort's idea). More broadly, "split the problem in half, solve each half, combine" is one of the first strategies experienced competitive programmers try when a brute-force approach is too slow.
1959–1961 · Tony Hoare, England

Quicksort — Partitioning Around a Pivot

+
The Source
Tony Hoare developed quicksort: choose an element (the "pivot"), rearrange ("partition") the array so that all elements less than the pivot come before it and all elements greater come after it (with the pivot now in its final sorted position), then recursively apply the same process to the sub-arrays on each side of the pivot.
Vocabulary
Pivotan element chosen (by some rule — e.g., the first element, the last, or a random one) around which the rest of the array is rearranged during partitioning.
Partitionthe process of rearranging an array so that all elements smaller than the pivot are on one side and all elements larger are on the other, with the pivot between them.
Average casethe expected behavior of an algorithm over "typical" or randomly-distributed inputs, as opposed to the absolute worst possible input. Quicksort's average case is excellent even though its worst case is poor.
Adversarial inputan input specifically constructed to trigger an algorithm's worst-case behavior — relevant in competitive programming, where problem setters sometimes deliberately include such inputs ("anti-test cases") to fail naive implementations.
First Principles

Quicksort is also divide-and-conquer, but it divides the work differently from merge sort — and this difference is illuminating. Merge sort does the "hard work" (merging) after the recursive calls — the recursive calls produce sorted halves, and then merging combines them. Quicksort flips this: it does the "hard work" (partitioning) before the recursive calls — partitioning produces two halves with a guarantee (everything on the left is ≤ pivot, everything on the right is ≥ pivot), and then nothing further is needed to combine them — once both halves are individually sorted, the whole array is sorted, because of the guarantee partitioning already established.

After partitioning around pivot 3, everything ≤3 is to its left and everything ≥3 is to its right, AND the pivot is in its final position. No merge step needed — just recursively sort each side.

This raises an obvious question: how good the split is depends entirely on how good the pivot choice is. If the pivot happens to be the median (the middle value) of the array, partitioning splits it into two roughly equal halves — and you get the same "halve each time" behavior as merge sort, giving O(n log n). But if the pivot is consistently the smallest or largest element (e.g., if you always pick the first element as pivot, and the array happens to already be sorted), partitioning produces one side with 0 elements and the other with n−1 — barely dividing the problem at all. In that case, the recursion has n levels instead of log(n), and the total work becomes O(n²) — the same bad complexity as insertion sort's worst case.

This worst case is the origin of the "average case" vs. "worst case" distinction becoming practically important, not just theoretical. Quicksort's average case (over random pivot choices, or random input order) is O(n log n) with very good real-world constants — often faster in practice than merge sort despite having the same Big-O class — but its worst case is O(n²), and that worst case can be deliberately triggered by an adversary who knows your pivot-selection rule. This is precisely why competitive programming problems sometimes include "anti-quicksort" tests, and why production-quality implementations often choose the pivot randomly or use a more sophisticated rule — making it much harder for any single input to reliably trigger the worst case.

There's a broader lesson here that recurs throughout algorithm design: an algorithm's Big-O classification can hide important practical differences. Merge sort and quicksort are "both O(n log n)" in the sense that matters for the worst case vs. average case distinction, but they behave very differently in practice, have different memory needs (quicksort can be done in-place; standard merge sort needs auxiliary space), and respond differently to adversarial inputs. Reading Big-O as the complete picture, rather than a useful first-pass summary, is a common early mistake.

Why this matters for CP The partitioning idea — "rearrange so everything on one side satisfies property X and everything on the other doesn't, using a single pass" — is the core of the "two pointers" / "Dutch national flag" technique, used constantly for in-place array manipulation problems. Additionally, a variant of quicksort's partitioning, called "quickselect," can find the k-th smallest element in an array in O(n) average time without fully sorting it — a useful trick when a problem asks for a specific rank/percentile rather than a full ordering.
1946 (first published), widespread by 1960s

Binary Search — The Simplest Divide-and-Conquer Algorithm

+
The Source
Binary search finds a target value within a sorted array by repeatedly comparing the target to the middle element of the current range: if equal, done; if the target is smaller, restrict the search to the left half; if larger, restrict to the right half — repeating until the target is found or the range becomes empty.
Vocabulary
Search spacethe set of possible answers or positions that haven't yet been ruled out — binary search's defining feature is halving this space with every comparison.
Invarianta condition that remains true throughout an algorithm's execution, no matter how many steps have happened — for binary search, "if the target exists, it's within the current [low, high] range" is an invariant that holds before and after every step. Invariants are the basis for proving an algorithm is correct.
Off-by-one errora very common bug where a loop or range boundary is set to one position too many or too few (e.g., using < instead of <=). Binary search is notorious for these errors precisely because it deals with boundaries (low, high, mid) on every step.
First Principles

Binary search is, in some sense, the "purest" example of the eliminate-half-the-possibilities idea that's been a thread throughout this entire timeline — Egyptian doubling, the binary search tree, even the underlying intuition of the Sieve. Here it appears in its most distilled form, applied directly to a sorted array rather than a tree structure.

Why does binary search require the array to be sorted? Because the entire algorithm hinges on a single deduction: "if the middle element is greater than the target, the target (if present) cannot be anywhere in the right half." This deduction is only valid if you know everything in the right half is ≥ the middle element — which is exactly what "sorted" guarantees. This is a clean example of a recurring theme: a precondition (the data being sorted) unlocks an algorithm (binary search) that would be invalid without it — and the cost of sorting (O(n log n), from this era's earlier entries) is often worth paying once if it enables many subsequent O(log n) searches.

The "invariant" concept deserves attention here because binary search is the canonical example for learning to reason about invariants, and this skill transfers to nearly every algorithm you'll write. The invariant — "if the target exists in the array, it exists within the current [low, high] range" — is true at the very start (the range is the whole array) and must remain true after every step. Checking "does my update to low/high preserve this invariant?" is exactly how you verify binary search is correct, and exactly how you debug it when it isn't. Off-by-one errors are so common here precisely because there are several superficially-similar-but-subtly-different ways to update low and high, and only some of them preserve the invariant correctly in all cases.

Binary search's reach extends far beyond "find a value in a sorted array." A whole category of problems — sometimes called "binary search on the answer" — applies the same halving logic not to find a value's position, but to find the smallest or largest value of some quantity for which a yes/no condition holds (e.g., "what's the minimum possible maximum load, such that the work can be split among k workers within that load?"). The key requirement is the same kind of invariant: the yes/no answer to "is X achievable?" must be monotonic — once it becomes true (or false) as X increases, it stays that way. Recognizing this monotonicity in a problem that doesn't superficially look like "searching an array" is one of the more satisfying "aha" moments in intermediate competitive programming.

The shape of this era, looking back Insertion sort (O(n²), the honest baseline), merge sort (O(n log n) via divide-then-combine), quicksort (O(n log n) average via partition-then-divide, with a notable worst case), and binary search (O(log n), the purest expression of halving). Together, these four algorithms — all from roughly the same 15–20 year window — define the vocabulary ("divide and conquer," "pivot," "invariant," "worst case vs. average case") that essentially all subsequent algorithm design builds on.
